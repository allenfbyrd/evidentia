# Evidentia v0.10.1 — implementation plan

> **Status**: PLANNED (drafted 2026-05-22 as Step 5.B of the v0.10.0
> pre-release-review). Sequenced as the v0.10.x consolidation pass
> immediately following v0.10.0 (the v0.10.x line opener — see
> [`v0.10.0-plan.md`](v0.10.0-plan.md)).
>
> **Theme**: complete the OCSF integration line + close the v0.10.0
> trust-boundary follow-up + extend the pilot-collector pattern to
> the remaining ~11 collectors.

---

## 1. Context

v0.10.0 shipped the OCSF-aligned findings schema, the bidirectional
OCSF Compliance Finding mapping layer, SARIF 2.1.0 output for
`evidentia gap`, and 3 pilot collectors (AWS, GitHub, Postgres)
populating the new `compliance_status` + `remediation` fields. Three
items were explicitly deferred to v0.10.1; one new LOW finding was
opened in the v0.10.0 pre-release-review that targets v0.10.1 for
remediation. This plan turns the deferral list into deliverables.

## 2. Phase 1 — Close F-V100-L1 (trust-boundary on `unmapped["evidentia"]`)

`finding_from_ocsf` reconstructs the original `SecurityFinding` from
the `unmapped["evidentia"]` block when present, trusting it as
authoritative. v0.10.0 only round-trips Evidentia-produced findings,
so the trust boundary is internal. v0.10.1 introduces the OCSF
*ingestion* collector (Phase 2 below) which will accept third-party
input — the trust boundary then becomes external and the block
cannot be honored unconditionally.

**Deliverable**: refactor `finding_from_ocsf` to take an explicit
`trust_unmapped: bool` parameter (default `True` for the
Evidentia-internal call path; `False` for the ingestion collector
path). When `False`, the unmapped block is ignored and the finding
is reconstructed from native OCSF fields only. The ingestion
collector chooses based on whether the OCSF source has a verifiable
origin (Sigstore signature on the artifact, GPG-signed envelope,
trusted SaaS-API source). Document the API change in
`docs/ocsf-mapping.md` §6 (round-trip fidelity) and
`docs/api-stability.md` (additive parameter with default — non-
breaking).

Acceptance gate: a Pydantic-validating but origin-forged OCSF doc
cannot impersonate Evidentia internal fields when ingested via the
collector path.

## 3. Phase 2 — Third-party OCSF ingestion collector

The deferred-from-v0.10.0 collector that reads OCSF JSON from a
file or API and produces `SecurityFinding[]`. Two paths:

- **Compliance Finding path** — uses `finding_from_ocsf(..., trust_unmapped=False)` from Phase 1.
- **Detection Finding path** — NEW. Prowler and AWS Security Hub
  emit OCSF Detection Finding (`class_uid` 2004), not Compliance
  Finding. The collector must recognize the class, map Detection
  Finding fields to `SecurityFinding`, and synthesize
  `compliance_status` from the detection-finding `finding_info` +
  `severity_id`.

Sub-deliverables:

- New `evidentia_collectors/ocsf/collector.py` — file + URL ingest
  modes; trust-boundary aware.
- Extend `evidentia_core.ocsf.finding_mapping` with a
  `finding_from_ocsf_detection(ocsf: dict) -> SecurityFinding`
  function. Append the Detection Finding field map to
  `docs/ocsf-mapping.md`.
- `evidentia collect ocsf --input <file-or-url>` CLI verb.
- Fixture corpus: 5–10 real Prowler outputs + 5–10 real AWS Security
  Hub OCSF outputs (sanitized — verify no real account IDs).

Acceptance gate: a Prowler-emitted OCSF file ingests cleanly into a
SecurityFinding list with the correct severity, compliance_status,
resource_id, and remediation fields populated; round-trip back to
OCSF preserves the source identity.

## 4. Phase 3 — Migrate the remaining ~11 collectors

Extend the v0.10.0 Phase 3 pilot pattern to every other collector:
Okta, the other 4 SQL adapters (MySQL, MSSQL, SQLite, Oracle),
Databricks, Snowflake, Vanta, Drata, BitSight, SecurityScorecard,
RiskRecon, UpGuard. At each `SecurityFinding(...)` construction
site:

- Set `compliance_status` explicitly per finding semantics.
  Vendor-risk collectors (Vanta / Drata / BitSight / SecurityScorecard /
  RiskRecon / UpGuard) — most findings are informational inventory,
  so `UNKNOWN` is the right default; only flag actual gaps as
  `FAIL`.
- Populate `remediation` where the source system supplies it
  (Snowflake security advisor recommendations, Vanta + Drata
  remediation text, Okta MFA-enrollment instructions, etc.).

Each collector gets a regression test for the new fields, mirroring
the v0.10.0 Phase 3 test additions for the AWS / GitHub / Postgres
pilots.

## 5. Phase 5 — Harden `scripts/bump_version.py` against third-party pin over-bumping

v0.10.0 pre-release-review Step 6 caught a release-blocker bug
(F-V100-M1, MEDIUM): `bump_version.py --to 0.10.0` substituted the
`py-ocsf-models>=0.9.0,<0.10.0` pin to `>=0.10.0,<0.11.0`, treating
the third-party version range as if it were an inter-package pin.
The bumped pin was unsatisfiable (no `py-ocsf-models 0.10.0`
exists) and would have broken any fresh install; caught by
`uv sync` during the pre-tag gate and reverted manually before tag.

The script's substitution logic must distinguish inter-package pins
from third-party pins. Two clean options:

1. **Explicit Evidentia-package allowlist** — substitute only pins
   whose package name is in a hard-coded `EVIDENTIA_PACKAGES` set.
   Simple; requires the list to stay current as packages are added.
2. **Use `[tool.uv.sources]` as the source of truth** — read the
   root `pyproject.toml`'s `[tool.uv.sources]` to identify workspace
   members (entries marked `{ workspace = true }`), substitute only
   those pins. Self-documenting via the existing manifest; no
   maintenance.

Recommend option 2.

**Acceptance gate**: after the fix, `bump_version.py --to <next>`
(any minor / major bump) does NOT change `py-ocsf-models`,
`pydantic`, `boto3`, or any other third-party version pin. Add a
unit test that fixtures a synthetic mini-monorepo with mixed
workspace + third-party pins, runs the bumper, and asserts only
the workspace pins moved.

**Urgency**: NOT urgent for v0.10.1 itself — patch bumps
(0.10.0 → 0.10.1) don't change the inter-package pin range and so
don't re-trigger the bug. The bug recurs at the next minor (v0.11.0
or v1.0.0). Fix in v0.10.1 to land it before then.

## 6. Phase 4 — Optional follow-ups (capacity-permitting)

- `SecurityFinding` → `Finding` class rename via the
  `docs/api-stability.md` deprecation policy (keep `SecurityFinding`
  as a re-export alias for ≥1 minor cycle per the v0.10.0 frozen-
  models entry).
- `evidentia collect --format ocsf` output mode — emit collected
  findings directly as an OCSF Compliance Finding bundle (instead
  of only via the gap-report pipeline).
- A small `evidentia_core.audit.events.COLLECT_OCSF_EMITTED`
  EventAction (append-only — minor-safe) if OCSF export becomes a
  CLI surface.

## 6. Sequencing

```
Phase 1 (F-V100-L1 close) ─> Phase 2 (ingestion collector) ─> Phase 3 (~11 collector migration)
                                                              └─> Phase 4 (optional)
```

Phase 1 must land before Phase 2 (the ingestion collector depends
on the `trust_unmapped=False` API surface). Phases 3 + 4 are
independent and can land in parallel with each other.

## 7. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Detection Finding semantics are subtler than Compliance Finding | Spike against the real Prowler + AWS Security Hub corpus first; document the mapping in `docs/ocsf-mapping.md` before writing the collector. |
| Vendor-risk collectors emit inventory not failures — wrong `compliance_status` defaults could misrepresent posture | The v0.10.0 design already chose `UNKNOWN` as the safe default; v0.10.1 collector migration explicitly sets `UNKNOWN` for inventory findings + `FAIL` for actual gaps. Per-collector review. |
| Collector migration adds breadth without depth | Hold the line: the goal is field-level enrichment per the v0.10.0 Phase 3 pattern, not collector rewrites. Each collector migration should be a small diff. |
| Origin-verification design for trust-boundary close adds complexity | Lean on existing primitives: Sigstore bundle verify for files, signed-SaaS-API-source allowlist for URL ingest. No new crypto. |

## 8. Out of scope (deferred further)

- MCP-as-backend + GRC Engineering Club marketplace plugin (per
  `docs/ROADMAP.md` v0.10.x themes — later v0.10.x).
- Persona modes (auditor / engineer / TPRM).
- YAML-driven catalog / control-tier definitions.
- AI-governance regulatory packs (EU AI Act Annex IV, ISO/IEC 42001).
- Map onto OpenSSF Gemara reference model.

These remain v0.10.x line themes but do not gate v0.10.1.

follow-through on a single shipped line; pre-1.0 versioning rule
in `~/.claude/CLAUDE.md` reserves minor bumps for new feature
