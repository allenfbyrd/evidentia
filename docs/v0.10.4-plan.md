## 2. Polish items carried from v0.10.3 /code-review

### P1 — Module-level docstring choke-point note (loader.py)

- **Source**: v0.10.3 `/code-review` suggestion #1.
- **Scope**: One-line note in `evidentia_core.catalogs.loader`'s
  module docstring stating that **all** catalog loaders MUST go
  through `_load_catalog_data` — don't add a sibling `json.loads`
  or `yaml.safe_load` call elsewhere. Hardens the choke point
  against future-contributor drift.
- **Effort**: Trivial (1-line docstring edit).
- **Risk**: None.

### P2 — Error-message polish for no-extension catalog paths (loader.py)

- **Source**: v0.10.3 `/code-review` suggestion #2.
- **Scope**: When `_load_catalog_data` hits the unsupported-extension
  branch with `suffix == ''` (operator drag-and-drop without an
  extension), mention the extension allowlist explicitly in the
  error message. Current message is clear but doesn't tell the
  operator the fix is "rename to .yaml or .json".
- **Effort**: Trivial (1-line error-message edit).
- **Risk**: None.

### P3 — `framework_id` collision check in regenerate_manifest.py

- **Source**: v0.10.3 `/code-review` suggestion #3.
- **Scope**: After `scan_dir` builds its `entries` list for a tier
  directory, assert no two entries share the same `framework_id`.
  Today's `TIER_DIRS` config makes a collision extremely unlikely
  (would require a deliberate same-name `foo.json` + `foo.yaml`
  pair), but defensive guard is cheap and would catch a
  contributor who forgets to delete the JSON when converting to
  YAML.
- **Effort**: Small (~10 LOC + 1 test).
- **Risk**: None — failure mode is a clear `ValueError` at
  manifest-regen time, before the manifest ships.

### P4 — Multi-line round-trip test coverage (test_yaml_loader.py)

- **Source**: v0.10.3 `/code-review` suggestion #4.
- **Scope**: Extend `test_yaml_and_json_load_to_identical_catalogs`
  to cover controls with multi-line `description`, populated
  `assessment_objectives` and `parameters`. The current 7-control
  iso-27017-2015 stub uses single-line strings, so byte-equivalent
  round-trip is asserted indirectly via the registry test. A
  targeted multi-line round-trip would harden the proof for the
  first contributor who hand-authors a multi-line YAML catalog
  (the use case YAML is intended to enable).
- **Effort**: Small (~30 LOC test + fixture).
- **Risk**: None — pure additive test coverage.

### P5 — Pre-tag CHANGELOG-presence check (release.yml or pre-release-review skill)

- **Source**: v0.10.3 SHIP INCIDENT. `release.yml` first-fire on tag
  `v0.10.3 @ 487e0ac` succeeded at PyPI publish (7 wheels with PEP
  740 + SLSA) but FAILED at the `Extract release body from CHANGELOG`
  step — I missed adding the `[0.10.3]` block to `CHANGELOG.md`
  during Step 6 of the v0.10.3 pre-release-review. Downstream skip
  blocked GitHub Release + GHCR container publishing. Recovery
  required a Plan A move-tag re-fire after backfilling the block.
- **Scope**: Add a pre-tag gate so this cannot recur. Two viable
  layers (do both):
  1. **Skill layer** — `~/.claude/skills/pre-release-review/SKILL.md`
     Step 6 prep checklist gains "run `python scripts/extract_changelog_block.py $VERSION` locally; refuse to push tag if output is < 1500 bytes or extraction fails." Adds to the 16-row pre-push gate as row 17 (or replaces an N/A row when possible).
  2. **Workflow layer** — add a pre-publish workflow step (or a new
     `verify.yml` workflow on `push:branches:main`) that runs
     `extract_changelog_block.py` for the version derived from the
     latest workspace `pyproject.toml`, asserts the block exists +
     meets the size gate. Catches the same drift in CI before the
     tag fires.
- **Effort**: Small (~15 LOC skill prose + ~15 LOC workflow step).
- **Risk**: None — pure additive gate. Failure mode: a clear refusal
  to push the tag locally, OR a CI failure on `main` before the tag
  ever fires.
- **Related**: the underlying `extract_changelog_block.py` script
  already enforces the 1500-byte sanity gate at workflow runtime;
  P5 lifts that check to pre-tag time so the gate failure is loud
  earlier and recovery is cheaper (no need to move a tag).
