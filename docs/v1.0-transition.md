## What v1.0 stands for (working theme candidates)

The 2026-04-28 §10 Q4 lock-in reserved v0.9.0 for
**federal-compliance theme**: CONMON cycle calendar + OSCAL
POA&M emit + walk-through opener with a domain expert.

v1.0 itself sits AFTER v0.9.0 in the natural sequence. Two
candidate theme framings, each pulled forward to the v1.0
narrative:

### Candidate A — "Federal compliance shipped"

v1.0 = "v0.9.0 federal-compliance theme delivered + tested
against real federal-SI workflows + accepted by an external
domain expert". Concretely:

- CONMON cycle calendar shipped (read-only library + CLI
  listing upcoming monthly / quarterly / annual control
  cadence per framework)
- OSCAL POA&M emit shipped (auto-generation from gap-analysis
  findings; manageable POA&M tracking primitives — state
  model + milestones + cycle-diff)
- Domain-expert walk-through completed; capability-matrix.md
  expanded with real-world scenario rows
- 1+ federal-SI partner has run the v0.9.0 surface against
  their actual customer scenarios + provided feedback

### Candidate B — "API stability commitment"

v1.0 = "Public API contract is stable; semantic-versioning
guarantees apply". Concretely:

- Pydantic models in `evidentia_core.models.*` are public
  schema (frozen field names + types; non-breaking field
  additions only between v1.x releases)
- `EventAction` enum is public schema (additions only)
- CLI flags are public schema (deprecation cycles for
  breaking changes)
- Plugin contracts (`AuthProvider`, `StorageBackend`,
  `MarketplaceProvider`, `BaseSaaSCollector` shipped v0.8.0
  P0.4) are stable
- Library entry points (`evidentia_core.gap_analyzer.GapAnalyzer`,
  `evidentia_ai.risk_statements.RiskStatementGenerator`,
  `evidentia_ai.eval.DFAHarness`, etc.) follow the same
  posture

Both candidates are non-mutually-exclusive. Most likely v1.0
combines them: ship the federal-compliance theme (Candidate A)
+ commit to the public API surface (Candidate B) at the same
time.

## What v1.0 will NOT do

The Allen-confirmed standing direction (per `~/.claude/CLAUDE.md`
+ the §0 absolute-secrecy posture re-confirmed multiple times
across the v0.7.x → v0.8.x cycle):

- **No commercialization vocabulary** on the public surface.
  v1.0 ships as OSS only; no public references to paid /
  Pro / Enterprise tier.
- **No GUI v2.** Web UI alpha.2 (v0.7.6) is the OSS UI
  posture. Surface upgrades happen via the existing alpha.2
  path; no separate "v1.0 UI" release.
- **No multi-tenant SaaS.** Per the standing rule + the
  threat models documented across the v0.8.x line, multi-
  tenant deployments are operator-responsibility (CIMD
  metadata + transport auth they wire); Evidentia itself
  doesn't host.
- **No hosted federal-cloud variant.** v0.9.0 federal-compliance
  ships the library + CLI primitives; FedRAMP-authorized
  hosting is downstream / out of scope for v1.0.

## API-stability commitments (Candidate B detail)

If v1.0 commits to API stability, the contract is:

### Frozen surfaces

- **Pydantic model field names + types** in
  `evidentia_core.models.*`. Adding optional fields is
  non-breaking; renaming or removing fields is a major-bump
  trigger.
- **`EventAction` enum values**. Adding new EventActions is
  non-breaking; renaming or removing values is a major-bump
  trigger.
- **CLI flag names + semantics**. Adding new flags is
  non-breaking; deprecation cycles cover renaming or removal.
- **Plugin contracts** (`AuthProvider.__init__` signature +
  `StorageBackend[T]` ABC method signatures + etc.).
- **Public library entry points** (`evidentia_core.*` +
  `evidentia_ai.*` + `evidentia_collectors.*` +
  `evidentia_integrations.*` + `evidentia_api.*` +
  `evidentia_mcp.*`).

### Non-frozen surfaces

- **Internal helper functions** (underscore-prefixed +
  documented as private).
- **Test fixtures** (operators don't import these).
- **Bundled catalog content** (catalogs evolve as
  authoritative sources update; per-catalog version pinning
  via `evidentia catalog pin <framework> <version>` is the
  operator escape hatch).
- **Threshold defaults** (operators tune via CLI flags;
  default-changes are documented but non-breaking).

### Deprecation cycle

- **Deprecation warning**: at minimum 1 minor-release cycle
  before removal.
- **Deprecation log entry**: every deprecation announced in
  CHANGELOG + ROADMAP.
- **Removal**: in the v(N+2).0 minor or later, never within
  a v(N).x patch.
