## P0 — Frontend major bumps (PR #21 re-issue)

### P0.1 — Tailwind 3 → 4 migration

**Largest single bump in v0.7.14.** Tailwind 4 moved from JS
config (`tailwind.config.ts`) to CSS-first config (`@theme`
directive in CSS).

**Current state** (`packages/evidentia-ui/tailwind.config.ts`):
- Full shadcn/ui "New York" preset
- 12 custom colors using `hsl(var(--*))` CSS variables
- Severity palette (critical / high / medium / low /
  informational)
- Border radius variables
- Accordion keyframes + animations
- `tailwindcss-animate` plugin

**Migration steps**:

1. **Read tailwind 4 official migration guide**
   (https://tailwindcss.com/docs/v4-beta) to confirm the
   shadcn/ui preset has a v4-compatible counterpart.
2. **Verify `tailwindcss-animate` v4 compatibility** — the
   plugin may need a major bump or replacement.
3. **Rewrite `tailwind.config.ts` → CSS-first `@theme` block**
   in `packages/evidentia-ui/src/index.css`. Move all 12 custom
   colors + severity palette + border radius + keyframes +
   animations into `@theme {...}`.
4. **Verify build pipeline**: `npm run build` produces same
   dist output (compare bundle sizes pre/post; visual diff on
   the SPA across the 3 main routes — Gap Analyze, Gap Diff,
   Risk Generate).
5. **Update vite-plugin config** if needed (tailwind 4 has a
   first-class vite plugin that's preferred over the legacy
   PostCSS chain).
6. **Tests**: `npm run typecheck` + `npm run build` + `npm run
   test` (vitest) all green.

**Estimated effort**: 1-2 days for the careful migration. If it
exceeds 2 days, defer to v0.8.0 P5 (alongside the UI alpha.3
work that the AI moat features will eventually surface).

**Commit**: `chore(deps-dev): tailwind 3 → 4 (CSS-first config migration)`

### P0.2 — TypeScript 5 → 6 + eslint 9 → 10 + plugin bumps

**Smaller scope than P0.1; mostly mechanical**:

1. **TypeScript 5 → 6** — single known issue (per PR #21's
   failing CI):
   ```
   tsconfig.json(20,5): error TS5101: Option 'baseUrl' is
   deprecated and will stop functioning in TypeScript 7.0.
   Specify compilerOption '"ignoreDeprecations": "6.0"' to
   silence this error.
   ```

   Fix: add `"ignoreDeprecations": "6.0"` to `tsconfig.json`

   0.5** — version bumps; check release notes for breaking

4. **jsdom 25 → 29** + **postcss 8.5.10 → 8.5.14 (patch)** +
   **@types/node 22 → 25** — likely transparent; run vitest
   to catch any jsdom API-surface changes.

**Estimated effort**: 0.5-1 day.

**Commits** (one per major bump, per the plan §22.C pattern):

### P1.2 — Item 3 (Tableau Windows tempfile cleanup)

The current `contextlib.suppress(OSError)` on `tmp_path.unlink()`
is acceptable because Windows tempdir auto-cleanup picks up
leaked .csv files within the OS housekeeping cycle. v0.7.13
deferred a "proper fix" to v0.7.14.

**Approach**: replace `tempfile.NamedTemporaryFile(delete=False)
+ try/finally unlink` with a `tempfile.TemporaryDirectory()`
context manager. The Tableau SDK call uses the temp file's
path; the directory cleanup on context exit reliably removes
the file even on Windows.

**Estimated effort**: 0.25 day.

**Commit**: `fix(tableau): use TemporaryDirectory for Windows tempfile cleanup`

### P1.3 — Item 6 (Databricks LTS env-var extensibility)

The current `_CURRENT_LTS_RUNTIMES` frozenset requires manual
update each time Databricks releases a new LTS. v0.7.13 deferred
this as feature work paired better with v0.8.0 plugin contracts.

**Re-evaluation**: a small env-var-extensibility addition is
simpler than waiting for the full plugin contract. Reads
`DATABRICKS_EXTRA_LTS_RUNTIMES` (comma-separated runtime
versions) and merges into the frozenset.

**Approach**:
1. Add a module-level helper `_extra_lts_from_env() -> set[str]`
   that reads + validates the env var.
2. Merge `_CURRENT_LTS_RUNTIMES` ∪ `_extra_lts_from_env()` at
   call time in `_is_current_lts(runtime_version)`.
3. Test: 3 unit tests covering empty / single-value / multi-value.

**Estimated effort**: 0.25 day.

**Commit**: `feat(databricks): DATABRICKS_EXTRA_LTS_RUNTIMES env-var extensibility`
