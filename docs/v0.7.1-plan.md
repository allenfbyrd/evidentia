## Theme

**"AI features hardening + supply-chain polish."**

v0.7.0 hardened the collectors and signing pipeline to enterprise
grade; v0.7.1 brings the AI features (`evidentia-ai`) up to the same
bar, plus a handful of small supply-chain follow-ups identified
during the v0.7.0 RC iteration and Step-4 review.

This is a deliberately scoped "polish" release — none of the v0.8+
"OSS-native AI moat" features (DFAH determinism harness, MCP server,
DSE screenshot validator, etc. — see [positioning-and-value.md §13.2](positioning-and-value.md))
land here. Those need their own scoped cycle.

## P0 — AI features enterprise-grade hardening (the headline)

### Problem

v0.7.0 collectors got the full enterprise-grade treatment: typed
exception hierarchy, `@with_retry` for transient failures,
`CollectionContext` metadata on every output, ECS-8.11 structured
logging via `evidentia_core.audit.logger`, explicit `BLIND_SPOTS`
disclosure lists where applicable. The AI features (risk_statements/,
explain/) did not — they were last touched in v0.4.0 and skipped the
v0.7.0 hardening pass.

Concrete v0.7.0 findings (per [capability-matrix.md](capability-matrix.md)
§3 HIGH bucket):

- 2 BLOCKER B3 violations (bare `except Exception`) in
  `risk_statements/generator.py` lines 173, 227
- Zero unit tests for the entire `risk_statements/` module
- Stdlib `logging` instead of ECS structured logger
- No `@with_retry` for transient LLM failures
- No `CollectionContext`-equivalent metadata on generated artifacts

### 4 design decisions to make first

These benefit from focused thought, not rushed inclusion in v0.7.0.

| # | Decision | Options | Recommendation |
|---|---|---|---|
| D1 | **Context-metadata abstraction for AI outputs** | (a) Repurpose `CollectionContext` (poor semantic fit — risk statements are *generated*, not *collected*; no `collected_at`/`source_system_id`). (b) New `GenerationContext` type carrying `model`, `temperature`, `prompt_hash`, `run_id`, `generated_at`. (c) Inline metadata fields on `RiskStatement` directly. | **(b)** — clean abstraction, mirrors the collector pattern, lives in `evidentia_core.audit.provenance` next to `CollectionContext`. |
| D2 | **Retry strategy for LLM calls** | Instructor's `max_retries` handles JSON validation retries (response wasn't valid Pydantic); `@with_retry` from `evidentia_core.audit.retry` handles network/transient retries (5xx, timeout, rate-limit). They serve different purposes. | **Stack both**: `@with_retry` wraps the LLM call (network layer); Instructor's `max_retries=3` stays as-is (validation layer). Cap `@with_retry` at 3 attempts with exponential backoff. Cite the LiteLLM retry semantics in inline doc. |
| D3 | **EventAction additions for AI events** | New entries in `evidentia_core.audit.events.EventAction` enum. Naming should follow the existing `evidentia.<subsystem>.<verb>` pattern. | Add: `AI_RISK_GENERATED`, `AI_RISK_FAILED`, `AI_RISK_RETRY`, `AI_EXPLAIN_GENERATED`, `AI_EXPLAIN_FAILED`, `AI_EXPLAIN_RETRY`, `AI_EXPLAIN_CACHE_HIT`. Update `docs/log-schema.md` event catalog accordingly. |
| D4 | **Sigstore verify default policy** | Step-4 fix added a structured warning when `expected_identity`/`expected_issuer` not set (UnsafeNoOp fallback). v0.7.1 should consider stronger: emit a `_log.warning(action=VERIFY_SIGNATURE_FAILED)` event in addition to the report warning, OR change default to require identity (breaking API change). | **Emit the structured log warning** in v0.7.1 (additive, non-breaking). Defer the breaking-change require-identity default to v1.0 with a deprecation cycle. |

### Implementation outline

Once D1-D4 are decided:

1. **Add `GenerationContext` to `evidentia_core.audit.provenance`** (~50 lines + tests)
2. **Add the 7 new `EventAction` enum entries** (~10 lines + 1 test entry per `test_events.py`)
3. **Refactor `risk_statements/generator.py`**:
   - Add typed exception hierarchy: `RiskStatementError` → `LLMUnavailableError`, `RiskValidationError`, `RiskGenerationFailed`
   - Replace bare `except Exception` (lines 173, 227) with `except (LLMUnavailableError, RiskValidationError, RiskGenerationFailed) as e`
   - Wrap LLM call with `@with_retry(max_attempts=3)` from `evidentia_core.audit.retry`
   - Use `_log = get_logger("evidentia.ai.risk_statements")` with structured event emission
   - Attach `GenerationContext` to every `RiskStatement` output
4. **Refactor `explain/generator.py`** with the same pattern
5. **New `tests/unit/test_ai/test_risk_statements.py`** (~250-300 lines):
   - Mock `instructor.Instructor.chat.completions.create` via `unittest.mock.patch`
   - Cover sync `generate`, async `generate_async`, `generate_batch`, `generate_batch_async`
   - Cover validation retry (Instructor max_retries hit then succeeds)
   - Cover network retry (`@with_retry` exponential backoff)
   - Cover air-gap refusal (`OfflineViolationError` for cloud models)
   - Cover empty `gaps` list, oversized `gaps` list (1000 entries with rate-limit simulation)
   - Cover `GenerationContext` propagation
6. **Update `tests/unit/test_ai/test_explain.py`** with the new pattern (additive, not breaking existing tests)
7. **Update `docs/enterprise-grade.md`**: clarify that B3 is now closed across BOTH collectors AND AI features

**Estimated scope**: ~600-800 lines of source + test changes. 1-2 weeks of focused work.

## Post-review additions (v0.7.1 dev branch, 2026-04-25)

After PR #1 + PR #2 of the P0 work landed on the development branch,
a second-pass GRC + cybersecurity adversarial review surfaced four
HIGH and two MEDIUM findings that are being folded into v0.7.1 (rather
than deferred to v0.7.2) because the same fix pattern applies to PR #3
(`explain/`):

- **H1**: `AI_RISK_FAILED` events now carry `run_id` so SIEM queries on
  `evidentia.run_id` surface failures alongside successes.
- **H2**: retry events emitted from inside `_invoke_llm_*` inherit
  `trace.id` from the run_id scope, so retry storms are attributable
  to the batch.
- **H3**: `GenerationContext.credential_identity` field added, populated
  by the new `evidentia_ai.client.get_operator_identity()` helper.
  Closes the NIST AU-3 "Identity" gap for AI-derived artifacts.
- **H4**: CHANGELOG documents the v0.7.0 → v0.7.1 deserialization
  break (same forward-compat property as v0.7.0's
  `SecurityFinding.collection_context` field).
- **M1**: `assert risk is not None` in `_invoke_llm_*` replaced with
  explicit `raise RuntimeError(...)` so the invariant survives `-O`.
- **M2 + M4**: dedicated `AI_RISK_BATCH_COMPLETED` /
  `AI_EXPLAIN_BATCH_COMPLETED` event actions for batch-summary
  emissions; outcome is `success` only when `succeeded == total`,
  `unknown` otherwise (per ECS spec).
- **L2**: `prompt_hash` surfaced in the `AI_RISK_GENERATED` event
  payload so SIEM operators can join log events to artifact-level
  GenerationContext blocks.
- **L4**: model-level test file `tests/unit/test_models/test_risk.py`
  added covering `generation_context` default-None, round-trip
  serialization, and v0.7.0 → v0.7.1 backward-compat deserialization.

The same fixes will be applied to PR #3 (`explain/generator.py`) so
the `explain/` subsystem ships v0.7.1 with the same audit-trail
guarantees.
