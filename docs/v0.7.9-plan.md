## Theme

**"Financial-sector regulatory-compliance overlay — TPRM module +
model risk management overlay + 7 new bundled catalogs."**

Every regulated US financial institution operates under a
well-defined regulatory stack: SR 11-7 (Federal Reserve Supervisory
Letter on Model Risk Management), OCC Bulletin 2011-12 (parallel
OCC guidance), FFIEC IT Examination Handbook (6 booklets covering
information security, audit, management, operations, business
continuity, outsourcing), OCC Bulletin 2013-29 (Third-Party
Relationships), FRB SR 13-19 (Vendor Risk Management), FFIEC
Cybersecurity Assessment Tool, NYDFS 500, Basel III Pillar 2
operational risk. v0.7.9 brings Evidentia into that stack as
**OSS-justified industry overlay capabilities**.

The two new top-level capability modules (`evidentia tprm` +
`evidentia model-risk`) are first-class additions to the CLI +
REST API — same pattern as the existing `gap` / `risk` / `collect`
/ `oscal` / `integrations` modules. The 7 new catalogs ship in the
existing 4-tier redistribution model (all at Tier A — public
domain regulatory publications).

This is the largest patch in the v0.7.x line. Realistic ship
target: 8-10 weeks after v0.7.8. The work spans ~3 distinct
subsystems (TPRM, model risk, catalog ingest); could be sequenced
as v0.7.9.0 (TPRM core) + v0.7.9.1 (model risk) + v0.7.9.2 (new
catalogs) if the cycle hits unexpected complexity.

## P0 — Third-Party Risk Management (TPRM) module

New top-level concept: `evidentia tprm`. Independent of the
existing `gap` / `risk` / `collect` flows; integrates with them.
Lives in a new workspace package `packages/evidentia-tprm` (or as
a sub-namespace `evidentia_core.tprm` — TBD at design time per
plugin-contract scaffolding from v0.8.0).

### P0.1 — Vendor inventory schema

Pydantic model in `evidentia_core.models.tprm.Vendor`:

| Field | Type | Notes |
|---|---|---|
| `id` | `str` (ULID) | Stable identifier |
| `name` | `str` | Vendor name |
| `type` | `Literal['saas', 'subservice_org', 'contractor', 'data_processor', 'cloud_provider', 'open_source']` | Taxonomy aligned to FFIEC + NIST 800-161 |
| `criticality_tier` | `Literal['critical', 'high', 'medium', 'low']` | Per FFIEC Vendor Management criticality model |
| `relationship_owner` | `str` | Internal owner email/identifier |
| `contract_start_date` | `date` | |
| `contract_end_date` | `date \| None` | |
| `last_due_diligence_review` | `date \| None` | |
| `next_review_due` | `date \| None` | Computed from criticality tier (annual/quarterly/etc.) |
| `regulatory_classification` | `list[Literal['custody', 'clearing', 'model', 'data_processor', 'critical_third_party']]` | Per applicable regulation |
| `fourth_parties` | `list[FourthParty]` | Disclosed sub-processors |
| `residual_risk_score` | `int` | 1-25 inherent × control = residual matrix |
| `notes` | `str \| None` | |
| `evidence_refs` | `list[EvidenceRef]` | Sigstore-signed evidence collected per vendor |

### P0.2 — Due-diligence questionnaire generator

Auto-generates SIG (Standardized Information Gathering) /
SIG-Lite / CAIQ (Cloud Security Alliance) format questionnaires.
Pre-fills vendor name + relationship details from inventory; leaves
control questions for the vendor to answer.

CLI: `evidentia tprm dd-questionnaire generate --vendor-id <id> --format {sig,sig-lite,caiq} --output <path>`

REST: `POST /api/tprm/vendors/<vendor_id>/dd-questionnaire`

Output formats: Excel `.xlsx` (preferred by most TPRM teams) +
CSV + JSON. Excel uses the standard SIG layout so vendor responses
can flow back into Evidentia via a corresponding `evidentia tprm
dd-questionnaire ingest` command.

### P0.3 — Concentration-risk reporting

Aggregate vendor exposure across configurable dimensions:

| Dimension | Source | Use case |
|---|---|---|
| Region | vendor metadata | Regulator concentration-risk thresholds |
| Cloud provider | inferred from collectors + vendor metadata | Cloud concentration risk per FFIEC + OCC guidance |
| 4th party | `fourth_parties` field | Identify single points of failure |
| Service category | vendor `type` | Diversification analysis |

CLI: `evidentia tprm concentration-report --by region,cloud-provider --threshold 0.30 --output report.html`

REST: `GET /api/tprm/concentration?by=region,cloud-provider`

Output: HTML report with sortable tables + configurable
concentration-threshold flags. Export-to-Excel via a `--format xlsx`
flag.

### P0.4 — Vendor change-detection collectors

Pre-built collectors for vendor-risk SaaS providers — ingest their
ongoing-monitoring posture changes so Evidentia's TPRM register
auto-updates without manual re-survey:

| Collector | Source | Auth |
|---|---|---|
| Vanta | Vanta Public API → vendor risk reports + control test results | OAuth token |
| Drata | Drata Public API → vendor inventory + monitoring status | OAuth token |
| BitSight | BitSight Security Ratings API → security ratings + ratings history | API key |
| SecurityScorecard | SSC API → company score + factor scores | API key |

Each collector follows the existing v0.7.0 collector pattern
(`CollectionContext`, `BLIND_SPOTS`, `@with_retry`, ECS-structured
logging). Output: `SecurityFinding` per vendor per source, mapped
to FFIEC Vendor Management + OCC 2013-29 controls.

### P0.5 — OSCAL TPRM model emit

Extend OSCAL Assessment Results to include vendor-attributed
evidence as `actor` references on observations + extend AR
back-matter to include the full vendor inventory as
`actor.party` resources.

Implementation: extend `evidentia_core.oscal.exporter` to accept
an optional `vendor_inventory` parameter; if present, embed
vendor data in AR back-matter + reference vendors from
observations via OSCAL `actor.uuid` references. Maintains
trestle-conformance.

## P0.6 — Model Risk Management overlay (`evidentia model-risk`)

New top-level capability module per SR 11-7 + OCC Bulletin 2011-12.

### P0.6.1 — Model inventory schema

Pydantic model in `evidentia_core.models.model_risk.ModelInventory`:

| Field | Type | Notes |
|---|---|---|
| `id` | `str` (ULID) | Stable identifier |
| `name` | `str` | Model name |
| `purpose` | `str` | Business purpose (per SR 11-7 §III.A "Conceptual Soundness") |
| `methodology` | `Literal['statistical', 'ml', 'rules_based', 'llm', 'expert_judgment', 'hybrid']` | Maps to validation requirements |
| `vendor_or_internal` | `Literal['internal', 'vendor']` | Internal models follow §III; vendor models follow §V |
| `vendor_id` | `str \| None` | Cross-link to TPRM vendor inventory |
| `tier` | `Literal['1', '2', '3']` | SR 11-7 criticality tier (Tier 1 = high-impact + complex) |
| `owner` | `str` | Internal model owner |
| `inputs` | `list[ModelInput]` | Data sources, transformations, lineage |
| `outputs` | `list[ModelOutput]` | Decisions affected, downstream consumers |
| `last_validation_date` | `date \| None` | |
| `validation_findings` | `list[ValidationFinding]` | Open + closed findings with severity |
| `next_validation_due` | `date \| None` | Computed from tier + last validation |
| `retirement_plan` | `str \| None` | Per SR 11-7 §III.C |
| `evidence_refs` | `list[EvidenceRef]` | Sigstore-signed evidence chain |

### P0.6.2 — Model documentation generator

Per SR 11-7 §III.A "Conceptual Soundness" + §III.B "Ongoing
Monitoring" + §III.C "Outcomes Analysis", generates model
documentation templates in Markdown + Excel formats:

CLI: `evidentia model-risk doc generate --model-id <id> --format {md,xlsx,oscal} --output <path>`

Output sections per SR 11-7:
- §III.A.1 Model purpose + intended use
- §III.A.2 Theory + design + data
- §III.A.3 Implementation + numerical methods
- §III.B.1 Outcomes analysis (back-testing + benchmarking)
- §III.B.2 Process verification (sensitivity + bias)
- §III.C Ongoing monitoring (performance + change management)
- Appendix A: Data lineage (auto-populated from `inputs` + the
  v0.7.7/v0.7.8 SQL/cloud-DW collectors if linked)
- Appendix B: Validation history (auto-populated from
  `validation_findings`)
- Appendix C: Linked evidence (auto-populated from
  `evidence_refs`)

### P0.6.3 — Validation report templates

Per SR 11-7 §III.D + Annex matrix, generates validation report
scaffolding for an independent model validator to fill in:

CLI: `evidentia model-risk validation-report --model-id <id> --findings findings.json --output report.md`

### P0.6.4 — AI-feature linkage

The Evidentia AI features (`evidentia risk generate`, `evidentia
explain`) ARE model invocations under SR 11-7. v0.7.9 wires every
risk-statement run to a model-inventory entry:

- Add new field `model_inventory_ref: str | None` on `RiskStatement`
- The existing `GenerationContext` (model, temperature, prompt_hash,
  run_id, evidentia_version) ALREADY captures SR 11-7 §III.A.3
  "implementation" evidence — v0.7.9 documents this mapping
- Sigstore-sign the `(risk_statement, model_inventory_ref)`
  link so the SR 11-7 audit trail is cryptographically verifiable
- New CLI: `evidentia model-risk link --risk-id <r> --model-id <m>`

## P1.5 — Risk-governance primitives (Three Lines of Defense + KRI/KPI/KGI + Open FAIR)

The financial-services regulatory stack rests on a small set of
risk-governance primitives that every regulated institution operates
under. v0.7.9 ships these as first-class Pydantic schemas + capability
modules so Evidentia stops being just a controls-catalog tool and
becomes a risk-governance substrate.

### G1 — Three Lines of Defense / Three Lines Model (IIA) capability

Per the Institute of Internal Auditors' Three Lines Model (2020
update of the original Three Lines of Defense framework):

- **First Line**: management + operational functions that own and
  manage risk (CISO / CIO / CDO equivalents in tech contexts)
- **Second Line**: independent risk + compliance oversight functions
  that provide effective challenge to the first line
- **Third Line**: internal audit providing independent assurance to
  governing bodies + senior management

**New schema**: `evidentia_core.models.governance.LineOfDefense` enum
+ `Owner` Pydantic model with a `line_of_defense` field. Every
control assignment + risk statement + finding can be tagged with
which line owns/oversees it. Reports stratify by line.

**Capability**: `evidentia governance lines-report` — produce a
matrix showing controls + risks + findings stratified by line of
defense, useful for audit-committee + board-level reporting.

### G2 — Effective Challenge primitives (2LOD evaluation workflow)

Effective challenge is the second line's regulatory obligation to
independently evaluate first-line risk assessments and push back
when standards aren't met. v0.7.9 ships primitives for documenting
+ tracking effective-challenge events:

- New schema: `evidentia_core.models.governance.EffectiveChallenge`
  with fields for the first-line assessment under review, the
  second-line evaluator, the challenge basis (citation to standard
  / framework / internal policy), the outcome (accepted / modified /
  escalated), and Sigstore-signed evidence of the evaluation
- CLI: `evidentia governance challenge log --first-line-assessment <ref> --basis "NIST 800-53 RA-3" --outcome modified --notes "..."`
- Automatic linkage to gap-analysis findings + risk statements so
  each finding carries the chain of effective challenges that
  shaped its remediation plan

### G3 — KRI / KPI / KGI taxonomy (forward-looking risk indicators)

Standard governance taxonomy:
- **KGI** (Key Goal Indicator) — measures goal attainment (lagging)
- **KPI** (Key Performance Indicator) — measures performance (current)
- **KRI** (Key Risk Indicator) — measures forward-looking risk (leading)

**New schema**: `evidentia_core.models.metrics.{KRI, KPI, KGI}`
Pydantic models with target-threshold + warning-threshold + breach-
threshold bands, owner, frequency (real-time / daily / weekly /
monthly / quarterly), data-source linkage to evidence collectors,
and current-value timestamp.

**Capability**: `evidentia metrics dashboard` — generate a metrics
dashboard (HTML + JSON-export) showing KRIs/KPIs/KGIs with band
status. Designed to be consumed by the v0.7.8 Tableau + Power BI
output integrations so risk officers see consistent metric
definitions across surfaces.

**Pre-built KRI templates** mapped to the new bundled catalogs:
- FFIEC IS booklet → KRIs for outstanding-vulnerability counts,
  patch-cycle SLAs, MFA-coverage percentage
- OCC 2011-12 → KRIs for model-validation-overdue count,
  model-failure-rate-by-tier
- FFIEC Vendor Management → KRIs for overdue-vendor-DD count,
  concentration-risk-by-region

### G4 — Open FAIR risk quantification module

Open FAIR (Factor Analysis of Information Risk) is The Open Group's
standard for quantitative risk analysis — converts qualitative
risk statements into probability distributions for loss frequency
+ loss magnitude.

**New module**: `evidentia_ai.risk_quantification` (uses
`evidentia-ai` package because it leans on the same LiteLLM +
Instructor stack for model-driven distribution-fitting).

**Capability**:
- `evidentia risk quantify --risk-id <r> --method open-fair`
  produces FAIR-format quantitative risk: Loss Event Frequency
  (LEF) distribution × Loss Magnitude (LM) distribution → Annualized
  Loss Expectancy (ALE) distribution
- Output formats: FAIR-U JSON (consumable by FAIR-U + RiskLens),
  Monte Carlo simulation results (10k iterations default), board-
  presentation summary tables
- AI-feature linkage: same `GenerationContext` provenance + SR 11-7
  model-inventory linkage as the existing `risk generate`

### G5 — Process-as-code governance workflows

Modern risk programs treat governance processes as code: workflows
defined declaratively, versioned, tested, auditable. v0.7.9 ships
a small process-as-code primitive for governance workflows:

- New schema: `evidentia_core.models.governance.ProcessDefinition`
  — YAML-defined workflow with steps, owners, artifacts produced,
  control mappings
- CLI: `evidentia governance process run --definition <yaml>`
  executes the process definition end-to-end, capturing Sigstore-
  signed evidence at each step
- Reference workflows shipped as starter templates:
  `examples/governance-workflows/` covering: vendor-onboarding-DD,
  model-validation, incident-classification, change-control-review,
  policy-exception-tracking
- Each starter workflow maps to controls in the FFIEC + OCC + SR
  11-7 + NYDFS + Basel III catalogs

## P2 — Audit chain-of-custody for financial data

Extend existing Sigstore-signed evidence with retention metadata
that financial-services regulators require.

### P2.1 — `CollectionContext` retention fields

Add new optional fields:

```python
class RetentionMetadata(BaseModel):
    retention_years: int  # 7 for SOX; varies for GLBA + banking SR letters
    retention_basis: str  # e.g., "SOX 302/404 §103(a)(7)"
    worm_backend: WORMBackend | None  # provider + bucket-policy info
    legal_hold: bool = False  # active legal hold suspends auto-purge
```

### P2.2 — WORM backend support

`WORMBackend` Pydantic model with provider-specific fields:
- AWS S3 Object Lock (governance + compliance modes)
- Azure Immutable Blob (time-based + legal-hold)
- Google Cloud Storage Bucket Lock
- On-prem WORM appliances (NetApp SnapLock, Dell EMC Centera) —
  metadata-only; cannot validate without OS access

### P2.3 — Retention-period CLI

CLI: `evidentia oscal retention-info <ar.json>` — reads
RetentionMetadata from an AR + reports retention status,
days-to-purge, applicable regulation citation.

## Strategic positioning notes

The v0.7.9 capability surface positions Evidentia distinctly in
the financial-services compliance tooling landscape:

- **Vendor SaaS GRC** (Vanta, Drata, AuditBoard, MetricStream,
  ServiceNow IRM): mostly framework-coverage + workflow tools,
  do NOT ship SR 11-7 / OCC 2011-12 / FFIEC catalogs as bundled
  out-of-the-box content
- **Open-source GRC ecosystem** (CISO Assistant, Eramba, Comp AI,
  compliance-trestle): primarily focused on NIST 800-53 / ISO
  27001 / SOC 2; the financial-services regulatory stack is
  underserved
- **Evidentia v0.7.9 positioning**: the only OSS GRC tool that
  ships TPRM + model risk + FFIEC + OCC + SR 11-7 as first-class
  bundled capabilities, with cryptographic evidence chain-of-custody
  + air-gap support. Apache 2.0 license unlocks SI / MSP
  distribution paths.

This is **public-document positioning** that will be reflected
in the v0.7.9 ship's `docs/positioning-and-value.md` Q3-or-Q4 2026
re-sync (not in this plan; lands as part of the standard cadence
work).
