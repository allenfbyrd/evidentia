## Theme

**"Financial-sector regulatory-compliance overlay — TPRM module +
model risk management overlay + 7 new bundled catalogs."**

Every regulated US financial institution operates under a
well-defined regulatory stack. In April 2026 the Fed + OCC issued
**SR 26-02 + OCC Bulletin 2026-13a** as the active model-risk
guidance, superseding the 2011 originals (SR 11-7 + OCC Bulletin
2011-12). The 2026 guidance EXPLICITLY EXCLUDES generative AI and
agentic AI from scope: *"Generative AI and agentic AI models are
novel and rapidly evolving. As such, they are not within the scope
of this guidance."* Banks running LLM-driven controls today have
no governing regulatory framework — agencies have promised a
future RFI but no timeline. This v0.7.9 model-risk overlay
addresses both the current SR 26-02 / OCC 2026-13a stack AND the
intentional generative-AI gap (every Evidentia AI invocation traces
back to a model-inventory entry via the `GenerationContext`
provenance chain — see §6 and `docs/positioning-and-value.md`
§4.6 + §7 #8 for the SR-replacement-framework opportunity).

The full referenced stack: SR 26-02 (Federal Reserve Supervisory
Letter on Model Risk Management — supersedes SR 11-7), OCC Bulletin
2026-13a (parallel OCC guidance — supersedes OCC 2011-12), FFIEC IT
Examination Handbook (6 booklets covering information security,
audit, management, operations, business continuity, outsourcing),
OCC Bulletin 2013-29 (Third-Party Relationships), FRB SR 13-19
(Vendor Risk Management), FFIEC Cybersecurity Assessment Tool,
NYDFS 500, Basel III Pillar 2 operational risk. v0.7.9 brings
Evidentia into that stack as **OSS-justified industry overlay
capabilities**.

The two new top-level capability modules (`evidentia tprm` +
`evidentia model-risk`) are first-class additions to the CLI +
REST API — same pattern as the existing `gap` / `risk` / `collect`
/ `oscal` / `integrations` modules. The 7 new catalogs ship in the
existing 4-tier redistribution model (all at Tier A — public
domain regulatory publications).

This is the largest patch in the v0.7.x line. Realistic ship
target: 8-10 weeks after v0.7.8. The work spans ~3 distinct
subsystems (TPRM, model risk, catalog ingest); could be sequenced
as v0.7.9.0 (TPRM core) + v0.7.9.1 (model risk) + v0.7.9.2 (new
catalogs) if the cycle hits unexpected complexity.

### P0.2 — Due-diligence questionnaire generator

**Status: SHIPPED 2026-05-03 first slice (commit pending push to origin/main).**

Auto-generates pre-filled DD questionnaires for a vendor in the
inventory. Pre-fills vendor metadata (name / type / criticality
tier / contract dates / region / regulatory classification /
4th-party disclosures) so the receiving vendor sees only control
questions, not blank metadata forms.

**CLI**: `evidentia tprm dd-questionnaire generate --vendor-id <id> --format {evidentia-generic,caiq-lite,sig,sig-lite} --output-format {json,csv} --output <path>`

**REST**: `POST /api/tprm/vendors/{vendor_id}/dd-questionnaire?format=<format>`

**Format catalogue**:

| Format | Status | License | Question count | Notes |
|---|---|---|---|---|
| `evidentia-generic` | SHIPPED | Apache-2.0 (Evidentia-original) | ~20 | FFIEC-aligned baseline. 9 domains: Governance, Access Control, Data Handling, Incident Response, Business Continuity, 4th-Party Risk, Personnel, Insurance, Compliance. |
| `caiq-lite` | SHIPPED | CC BY 4.0 (CSA CAIQ v4.0.3 derivative) | ~25 | Representative subset of CSA's 245-question CAIQ; covers all 17 CAIQ control domains. Operators wanting full CAIQ should use the planned `--from-template` BYO path. |
| `sig` | STUB | Shared Assessments (paywalled) | n/a | Errors today with a clear "Shared Assessments paywalled content" message. Future: `--from-template <licensed-xlsx>` BYO ingestion. |
| `sig-lite` | STUB | Shared Assessments (paywalled) | n/a | Same as `sig`. |

**Output formats**:

- **JSON** — full Pydantic `Questionnaire` model dump; machine-
  consumable for downstream tooling
- **CSV** — flat (one row per question + vendor-prefill header
  section + blank `vendor_response` column for vendor to fill);
  spreadsheet-pivot friendly without an `openpyxl` dep
- **XLSX** — deferred. Would require `openpyxl` (~3 MB binary
  deps). CSV covers the use case; xlsx revisits if customer
  demand surfaces

**Engine** lives in `evidentia_core.tprm.questionnaire`:
`Question`, `VendorPreFill`, `Questionnaire`, `QuestionnaireFormat`,
`generate_questionnaire`, `render_csv_questionnaire`,
`shipped_formats`. Packaged data ships under
`evidentia_core/tprm/data/questionnaires/*.json` (loaded via
`importlib.resources` so the JSON files survive zipimport in
the wheel).

**Tests**: 24 unit + 7 CLI integration + 6 REST integration =
37 new. 1410 → 1441 total passing.

**Future v0.7.9 P0.2 follow-up sub-slices**:

- `evidentia tprm dd-questionnaire ingest --questionnaire <path>` —
  load completed vendor responses back into Evidentia + correlate
  to vendor inventory via the questionnaire's UUID
- `--from-template <licensed-xlsx>` BYO ingestion path for SIG /
  SIG-Lite — operators with Shared Assessments membership supply
  their licensed XLSX, Evidentia pre-fills vendor metadata into

- XLSX output (would gate on a `[xlsx]` extra carrying

### P0.3 — Concentration-risk reporting

**Status: SHIPPED 2026-05-03 (commit pending push to origin/main).**

Aggregate vendor exposure across configurable dimensions:

| Dimension | Source on Vendor | Use case |
|---|---|---|
| `region` | `vendor.region` (new field; free-text geo / cloud-region label) | Regulator concentration-risk thresholds |
| `cloud-provider` | direct `vendor.type==cloud_provider` PLUS `vendor.fourth_parties[].name` filtered to `type=cloud_provider` | Cloud concentration risk per FFIEC + OCC guidance |
| `4th-party` | `vendor.fourth_parties[].name` (any type) | Identify single points of failure |
| `service-category` | `vendor.type` | Diversification analysis |
| `criticality-tier` | `vendor.criticality_tier` | Operator scope-of-review prioritization |
| `regulatory-classification` | `vendor.regulatory_classification[]` (multi-valued per vendor) | OCC 2013-29 critical-third-party flagging |

CLI: `evidentia tprm concentration-report --by region,cloud-provider --threshold 30 --format html --output report.html`

REST: `GET /api/tprm/concentration?by=region,cloud-provider&threshold=30`

Threshold semantics: when set, every per-value distribution row whose
vendor share meets-or-exceeds the threshold gets `exceeds_threshold=true`
in the JSON shape and rendered with a flag column in the HTML output.
Operators get the concrete row to investigate.

Output formats:
- **HTML** — single-file (no external deps); sortable tables via inline
  click-to-sort JS; HTML-escapes all user-supplied vendor + value names
- **JSON** — full `ConcentrationReport` model dump; REST surface returns
  this shape natively
- **CSV** — flat (one row per dimension × value); columns:
  dimension / value / count / percentage / exceeds_threshold

`xlsx` export deferred — would require `openpyxl` or similar dep; CSV
covers the spreadsheet-pivot use case without bringing in the binary.

Engine + types live in the new sub-namespace
`evidentia_core.tprm.concentration` (`SUPPORTED_DIMENSIONS`,
`compute_concentration`, `ConcentrationReport`, `DimensionAnalysis`,
`ValueCount`, `render_html_report`, `render_csv_report`).

Tests: 20 unit + 6 CLI integration + 6 REST integration. 1377 → 1410
total passing.

Vendor model gained `region: str | None = None` field (free-text;
nullable; max-length 128). Existing stored vendor JSON files load
cleanly via Pydantic default-factory.

## P1.5 — Risk-governance primitives (Three Lines of Defense + KRI/KPI/KGI + Open FAIR)

The financial-services regulatory stack rests on a small set of
risk-governance primitives that every regulated institution operates
under. v0.7.9 ships these as first-class Pydantic schemas + capability
modules so Evidentia stops being just a controls-catalog tool and
becomes a risk-governance substrate.

### G1 — Three Lines of Defense / Three Lines Model (IIA) capability

Per the Institute of Internal Auditors' Three Lines Model (2020
update of the original Three Lines of Defense framework):

- **First Line**: management + operational functions that own and
  manage risk (CISO / CIO / CDO equivalents in tech contexts)
- **Second Line**: independent risk + compliance oversight functions
  that provide effective challenge to the first line
- **Third Line**: internal audit providing independent assurance to
  governing bodies + senior management

**New schema**: `evidentia_core.models.governance.LineOfDefense` enum
+ `Owner` Pydantic model with a `line_of_defense` field. Every
control assignment + risk statement + finding can be tagged with
which line owns/oversees it. Reports stratify by line.

**Capability**: `evidentia governance lines-report` — produce a
matrix showing controls + risks + findings stratified by line of
defense, useful for audit-committee + board-level reporting.

### G2 — Effective Challenge primitives (2LOD evaluation workflow)

Effective challenge is the second line's regulatory obligation to
independently evaluate first-line risk assessments and push back
when standards aren't met. v0.7.9 ships primitives for documenting
+ tracking effective-challenge events:

- New schema: `evidentia_core.models.governance.EffectiveChallenge`
  with fields for the first-line assessment under review, the
  second-line evaluator, the challenge basis (citation to standard
  / framework / internal policy), the outcome (accepted / modified /
  escalated), and Sigstore-signed evidence of the evaluation
- CLI: `evidentia governance challenge log --first-line-assessment <ref> --basis "NIST 800-53 RA-3" --outcome modified --notes "..."`
- Automatic linkage to gap-analysis findings + risk statements so
  each finding carries the chain of effective challenges that
  shaped its remediation plan

### G3 — KRI / KPI / KGI taxonomy (forward-looking risk indicators)

Standard governance taxonomy:
- **KGI** (Key Goal Indicator) — measures goal attainment (lagging)
- **KPI** (Key Performance Indicator) — measures performance (current)
- **KRI** (Key Risk Indicator) — measures forward-looking risk (leading)

**New schema**: `evidentia_core.models.metrics.{KRI, KPI, KGI}`
Pydantic models with target-threshold + warning-threshold + breach-
threshold bands, owner, frequency (real-time / daily / weekly /
monthly / quarterly), data-source linkage to evidence collectors,
and current-value timestamp.

**Capability**: `evidentia metrics dashboard` — generate a metrics
dashboard (HTML + JSON-export) showing KRIs/KPIs/KGIs with band
status. Designed to be consumed by the v0.7.8 Tableau + Power BI
output integrations so risk officers see consistent metric
definitions across surfaces.

**Pre-built KRI templates** mapped to the new bundled catalogs:
- FFIEC IS booklet → KRIs for outstanding-vulnerability counts,
  patch-cycle SLAs, MFA-coverage percentage
- OCC 2011-12 → KRIs for model-validation-overdue count,
  model-failure-rate-by-tier
- FFIEC Vendor Management → KRIs for overdue-vendor-DD count,
  concentration-risk-by-region

### G4 — Open FAIR risk quantification module

Open FAIR (Factor Analysis of Information Risk) is The Open Group's
standard for quantitative risk analysis — converts qualitative
risk statements into probability distributions for loss frequency
+ loss magnitude.

**New module**: `evidentia_ai.risk_quantification` (uses
`evidentia-ai` package because it leans on the same LiteLLM +
Instructor stack for model-driven distribution-fitting).

**Capability**:
- `evidentia risk quantify --risk-id <r> --method open-fair`
  produces FAIR-format quantitative risk: Loss Event Frequency
  (LEF) distribution × Loss Magnitude (LM) distribution → Annualized
  Loss Expectancy (ALE) distribution
- Output formats: FAIR-U JSON (consumable by FAIR-U + RiskLens),
  Monte Carlo simulation results (10k iterations default), board-
  presentation summary tables
- AI-feature linkage: same `GenerationContext` provenance + SR 11-7
  model-inventory linkage as the existing `risk generate`

### G5 — Process-as-code governance workflows

Modern risk programs treat governance processes as code: workflows
defined declaratively, versioned, tested, auditable. v0.7.9 ships
a small process-as-code primitive for governance workflows:

- New schema: `evidentia_core.models.governance.ProcessDefinition`
  — YAML-defined workflow with steps, owners, artifacts produced,
  control mappings
- CLI: `evidentia governance process run --definition <yaml>`
  executes the process definition end-to-end, capturing Sigstore-
  signed evidence at each step
- Reference workflows shipped as starter templates:
  `examples/governance-workflows/` covering: vendor-onboarding-DD,
  model-validation, incident-classification, change-control-review,
  policy-exception-tracking
- Each starter workflow maps to controls in the FFIEC + OCC + SR
  11-7 + NYDFS + Basel III catalogs

## P2 — Audit chain-of-custody for financial data

Extend existing Sigstore-signed evidence with retention metadata
that financial-services regulators require.

### P2.1 — `CollectionContext` retention fields

Add new optional fields:

```python
class RetentionMetadata(BaseModel):
    retention_years: int  # 7 for SOX; varies for GLBA + banking SR letters
    retention_basis: str  # e.g., "SOX 302/404 §103(a)(7)"
    worm_backend: WORMBackend | None  # provider + bucket-policy info
    legal_hold: bool = False  # active legal hold suspends auto-purge
```

### P2.2 — WORM backend support

`WORMBackend` Pydantic model with provider-specific fields:
- AWS S3 Object Lock (governance + compliance modes)
- Azure Immutable Blob (time-based + legal-hold)
- Google Cloud Storage Bucket Lock
- On-prem WORM appliances (NetApp SnapLock, Dell EMC Centera) —
  metadata-only; cannot validate without OS access

### P2.3 — Retention-period CLI

CLI: `evidentia oscal retention-info <ar.json>` — reads
RetentionMetadata from an AR + reports retention status,
days-to-purge, applicable regulation citation.
