## Theme

**"The OSS-native AI moat."**

v0.7.x closed the enterprise-grade gap (collectors, AI hardening,
supply-chain provenance) and brought every public capability to a
production-grade signal floor. v0.8.0 ships the differentiation
features that no commercial GRC vendor offers today:

- A **determinism harness** for AI-generated risk statements
  (DFAH — Decision-Faithfulness Assessment Harness — per
  [arXiv:2601.15322](https://arxiv.org/abs/2601.15322))
- **Policy Reasoning Traces** that record exact clause citations
  next to every claim (per
  [arXiv:2509.23291](https://arxiv.org/abs/2509.23291))
- An **MCP server** exposing Evidentia's library surface to AI
  agents (Claude / ChatGPT / Gemini / custom) under the standard
  Model Context Protocol
- A **stable plugin contract** so out-of-tree extension authors
  (community catalog providers, SI partners building custom
  integrations) can ship custom auth, storage, and marketplace
  backends without forking core

These capabilities sit on top of v0.7.x's signed-evidence + OSCAL
+ supply-chain story, not adjacent to it. The whole release is
designed so an auditor / 3PAO / federal SI can read the AR
artifacts, verify the determinism scores, and trace every claim
back to a policy clause without trusting an opaque LLM oracle.

This is a minor release, not a polish patch. Realistic ship target:
~3 months after v0.7.3.

## P0 — AI quality + ergonomics (the headline)

### P0.1 — DFAH determinism harness — `evidentia eval`

**What**: a new CLI subcommand and library entry point that
validates the `RiskStatementGenerator` and `ExplanationGenerator`
under DFAH framing. Three measurable dimensions:

| Dimension | Question | Failure surface |
|---|---|---|
| **Decision determinism** | Same `(input, model, temperature, prompt_hash)` → same statement (modulo whitespace + punctuation normalisation)? | Non-deterministic output; SIEM-side replay impossible |
| **Faithfulness** | Do the statement's claims trace back to the input control text + system context, or are they invented? | Hallucinated risk language reaches an audit report |
| **Replay equivalence** | Re-running the same `(input, model, temperature, prompt_hash, run_id)` produces output that hashes identically? | Audit-trail divergence between rebuilds |

**CLI shape (preview)**:

```bash
evidentia eval \
    --catalog nist-800-53-rev5-moderate \
    --samples 50 \
    --model gpt-4o-mini \
    --fail-on-determinism-rate-below 0.95 \
    --output /tmp/eval-report.json
```

**Architecture sketch**:

- New module `evidentia_ai/eval/` — `harness.py`, `metrics.py`,
  `seeds.py`. Reuses the v0.7.1 `GenerationContext` (which already
  captures `model`, `temperature`, `prompt_hash`, `run_id`).
- New `EventAction` entries: `AI_EVAL_STARTED`,
  `AI_EVAL_DETERMINISM_VIOLATION`,
  `AI_EVAL_FAITHFULNESS_VIOLATION`, `AI_EVAL_COMPLETED`.
- Eval reports survive Sigstore signing — the harness output IS
  audit evidence and an auditor running the harness independently
  can verify the publisher signed-and-attested the evaluation
  numbers they're reading.

**Why P0**: per
[`positioning-and-value.md` §11.3](positioning-and-value.md), no
commercial vendor offers this; it is the most defensible
auditor-facing claim Evidentia can make. Risk-register risk #3
(AI hallucination incident) is mitigated specifically by shipping
this in v0.8.

### P0.2 — Policy Reasoning Traces (PRT) mode

**What**: a new `--emit-trace` CLI flag (and equivalent
`emit_trace=True` kwarg on `RiskStatementGenerator.generate()`)
that adds an explicit, structured `reasoning_trace` field to each
generated risk statement.

```bash
evidentia risk generate \
    --inventory my-controls.yaml \
    --report gap.json \
    --emit-trace \
    --output risks-with-trace.json
```

Each trace is a list of (claim, clause-citations, confidence)
triples. Auditors can review whether the LLM's chain of reasoning
maps to the policy text the gap analysis was scoped against.

**Architecture sketch**:

- New Pydantic model `evidentia_core.models.risk.ReasoningTrace`
  with fields `claim` / `clause_citations` / `confidence`.
- Optional field on `RiskStatement` (default `None` for v0.7.x
  backward compat; tighten to required in a future release with a
  deprecation cycle).
- OSCAL emit extends back-matter to include the trace as
  `prop[name=evidentia.reasoning_trace,
   ns=https://evidentia.dev/oscal]`.
- New `EventAction`: `AI_RISK_TRACE_EMITTED`.

**Why P0**: makes Evidentia's AI features auditor-defensible.
Every commercial GRC tool generates risk statements as opaque
blobs; PRT-mode statements are traceable to source. Differentiator
for federal compliance + 3PAO use cases.

### P0.3 — MCP server — `evidentia mcp serve`

**What**: a new workspace package `packages/evidentia-mcp/` that
exposes Evidentia's library surface to AI agents via the standard
[Model Context Protocol](https://modelcontextprotocol.io/). Tools:

| Tool | Wraps | Auth model |
|---|---|---|
| `gap_analyze(inventory_path, frameworks)` | `evidentia_core.gap_analyzer.GapAnalyzer.analyze()` | none (read-only) |
| `risk_generate(gap_id, system_context)` | `evidentia_ai.risk_statements.RiskStatementGenerator.generate()` | optional API key |
| `explain_control(control_id, framework)` | `evidentia_ai.explain.ExplanationGenerator.generate()` | none |
| `gap_diff(report_a_path, report_b_path)` | `evidentia_core.gap_diff.compute_gap_diff()` | none |
| `oscal_export(gap_report_path)` | `evidentia_core.gap_analyzer.export_report(format="oscal-ar")` | optional Sigstore identity |
| `collect_*(provider_args)` | `evidentia_collectors.*` | provider creds via env |

**Architecture sketch**:

- New workspace member at `packages/evidentia-mcp/` mirroring the
  `evidentia-ai` package layout.
- Built on the official `mcp` Python SDK (or `fastmcp` if its
  server-side beta has stabilized at ship time).
- `evidentia mcp serve [--stdio | --http :PORT]` for the CLI
  surface; `evidentia mcp doctor` validates server config + tool
  registration.
- CIMD (Client ID Metadata Document) support for production
  deployments where the server needs to identify clients.
- Integration tested against ≥ 2 MCP clients (Claude Desktop /
  Claude Code / ChatGPT desktop) in `tests/integration/test_mcp/`.

**Why P0**: MCP is the standard for AI-agent ↔ tool integration;
shipping an MCP server lets any agent run Evidentia's full library
surface in plain language. The library-first architecture (every
capability is already a typed Python function with Pydantic
interfaces) makes the MCP wrapping mechanical — the marginal cost
is small relative to the user-facing value.

### P0.4 — Plugin-contract scaffolding

**What**: a documented, stable extension surface for three plugin
types so out-of-tree authors can ship their own implementations
without forking core.

| Plugin contract | OSS reference implementation in v0.8.0 | Use case it unlocks |
|---|---|---|
| `AuthProvider` | `evidentia_core.plugins.auth.local_token` (token file) | Single-tenant token auth on `evidentia serve` (the v0.4.0 token-auth foundation, formalised). |
| `StorageBackend` | `evidentia_core.plugins.storage.file_backend` (filesystem; already exists, completed to the contract) | Persist gap reports / collector findings somewhere other than the local filesystem (S3, IPFS, a federal-customer-specific NFS share, etc.). Designed in `Evidentia-Architecture-and-Implementation-Plan.md` Appendix D; v0.8.0 finishes the contract. |
| `MarketplaceProvider` | `evidentia_core.plugins.marketplace.local_directory` (filesystem) | Pull OSCAL catalogs from a community-hosted catalog mirror, an organization's private catalog registry, or a research-group's local-language catalog set, instead of (or alongside) the bundled 82-catalog set. |

**Architecture sketch**:

- New module `evidentia_core.plugins.{auth,storage,marketplace}`
  with abstract base classes + entry-point discovery via
  `importlib.metadata.entry_points(group='evidentia.plugins')`.
- Reference implementations ship in OSS under
  `evidentia_core.plugins.<type>.<impl>` so contributors have a
  working shape to copy.
- New documentation: `docs/extending.md` covering the contract,
  the entry-point declaration pattern, and the test fixtures for
  validating a custom plugin against the contract.
- Plugin discovery is opt-in (controlled by the runtime config);
  Evidentia's default behaviour is unchanged.

**Why P0**: SI partners building delivery contracts for federal
customers + community catalog mirror operators + security
researchers maintaining out-of-tree collectors all hit the same
wall today — they have to monkey-patch internals. A documented
plugin contract is cheaper to maintain than a growing list of
forks. The contract is generic; specific extensions ship outside
the OSS repo on the schedule that suits their author.

These close out enterprise-grade follow-ups still on the

Out of scope (each gets its own cycle or release theme):

  full auto-scoring capability stays research-grade.
- **CONMON (Continuous Monitoring) cycle calendar** — anchors the
  v0.9.0 "Federal compliance" theme alongside POA&M generation.
- **OSCAL POA&M emit + auto-generation** — v0.9.0.
- **Tauri desktop packaging** — out of scope; revisit when
  there's clear demand.

## Acceptance criteria

v0.8.0 is "shipped" when **all** of these pass:

- [ ] `evidentia eval` CLI ships and gates determinism + faithfulness on a default catalog with a measured pass-rate baseline reported in `CHANGELOG.md` (numerical baseline TBD post-implementation; aim ≥ 0.90)
- [ ] `evidentia risk generate --emit-trace` ships; OSCAL AR back-matter contains the trace; `compliance-trestle` round-trip test still green
- [ ] `evidentia mcp serve` ships with the documented tool surface; integration tests against ≥ 2 MCP clients pass in CI
- [ ] Plugin-contract scaffolding (AuthProvider, StorageBackend, MarketplaceProvider) ships with at least one OSS reference implementation per type + `docs/extending.md`
- [ ] Mutation-testing baseline ≥ 65% (G1)
- [ ] At least 5 hypothesis-driven tests on the crosswalk + normaliser (G2)
- [ ] Prometheus `/metrics` endpoint live on `evidentia serve` (G3)
- [ ] Reproducible-build target produces identical-hash wheels across two clean builds (G4)
- [ ] `docs/benchmarks.md` published with reproducible numbers (G7)
- [ ] `docs/evidence-integrity.md` published with anti-tamper deployment guidance (G8)
- [ ] OpenSSF Best Practices Badge **filed** before v0.8.0 ship; ideally rendered on the README
- [ ] All v0.7.x baselines hold: pytest 100% green; mypy strict clean; ruff lint clean; `pypi-attestations verify pypi` green; `gh attestation verify` green (the new SLSA-path verifier from v0.7.3 S3)
- [ ] Pre-release-review skill (Pre-tag variant) all 6 steps green
