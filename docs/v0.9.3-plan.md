## Phase 1 — CONMON daemon (Theme A)

Builds on `evidentia_core.conmon` (v0.9.0) and `/api/conmon/*` (v0.9.2). The library + REST surfaces are read-only and require operators to poll. Phase 1 makes Evidentia the actor that polls.

### P1.1 — `evidentia conmon watch --poll` daemon

NEW long-running CLI command. Reads a state file (slug → last_completed mapping), polls `derive_status()` on a configurable interval, emits `CONMON_CYCLE_DUE`/`CONMON_CYCLE_OVERDUE` events when state transitions occur.

Key design decisions:
- **Single-process, no fork**: Cron-style loop with `time.sleep()` inside the main asyncio event loop. Operators wanting daemonization use systemd / supervisord / cron — Evidentia stays in the foreground for AC/DR clarity.
- **Configurable poll interval**: `--poll-interval=<seconds>` flag, default 3600 (1 hour). Min 60s.
- **State persistence**: `--state-file <path>` for the slug→last_completed YAML; updated atomically when operator runs `evidentia conmon mark-completed <slug>` (also new in P1.1).
- **Graceful shutdown**: SIGINT/SIGTERM handler → finishes current poll cycle → flushes audit events → exits 0.
- **No durable queue**: State changes emit audit events synchronously; no message broker; no async result handling. Operator inspects audit log for history.

NEW module `evidentia_core.conmon.daemon` (~250 LOC + ~120 LOC tests).
NEW EventActions: `CONMON_DAEMON_STARTED`, `CONMON_DAEMON_STOPPED`, `CONMON_CYCLE_MARKED_COMPLETED`.

NEW `docs/conmon-daemon-deployment.md` (~150 LOC) — operator deployment guide covering 3 platforms:
- `evidentia-conmon.service` systemd unit (Linux — primary target)
- `com.evidentia.conmon.plist` launchd config (macOS)
- PowerShell registration script via `sc.exe` (Windows — federal Windows shops)
All marked as REFERENCE, not enforcement. Operators retain full control over their service manager choice.

### P1.2 — Basic alerting

Threshold/condition-based alerts delivered when daemon detects state transitions. Two channels:

- **Email (SMTP)**: config via `--smtp-host` / `--smtp-port` / `--smtp-from` / `--smtp-to` flags; STARTTLS only (no plaintext).
- **Generic webhook**: `--webhook-url <url>` POSTs JSON payload `{cadence, framework, activity, state, days_until_due, event_id}` with HMAC-SHA256 signature header.

**Credential handling (per global secret protocol)**:
- SMTP password: resolution precedence `--smtp-password-file <path>` > `EVIDENTIA_SMTP_PASSWORD` env > fail-fast with clear error
- Webhook secret: resolution precedence `--webhook-secret-file <path>` > `EVIDENTIA_WEBHOOK_SECRET` env > fail-fast
- **Explicitly REJECTED**: `--smtp-password <value>` or `--webhook-secret <value>` CLI flags (would leak via shell history + process lists). CLI arg parser raises hard error if seen.
- Rationale: file-based reference (`--*-file`) supports Kubernetes Secrets / HashiCorp Vault / AWS Secrets Manager / SystemD `LoadCredential`. Env-only excludes those operator patterns.

Alert deduplication: per-(slug, state) suppression for 24h to prevent spam (configurable via `--alert-suppression-hours`). State stored in the daemon's state file.

NEW module `evidentia_core.conmon.alerting` (~220 LOC + ~180 LOC tests; +30 LOC for file-based credential resolution + 2 tests covering file-based path).
NEW `evidentia_integrations` reference impl: `evidentia_integrations.alerting.smtp` + `evidentia_integrations.alerting.webhook`.
NEW EventAction: `CONMON_ALERT_DISPATCHED` + `CONMON_ALERT_SUPPRESSED`.

### P1.3 — Control health scoring

Aggregate metrics: pass/fail/stale per framework + per project. NEW CLI: `evidentia conmon health [--framework X] [--state-file Y] [--json]`. Stdlib-only (no Prometheus dependency); reuses `derive_status()` from v0.9.0 library.

Output shape:
```
| Framework             | Total | Current | Due-soon | Overdue | Stale |
|-----------------------|-------|---------|----------|---------|-------|
| nist-800-53-rev5      | 7     | 4       | 1        | 2       | 0     |
| fedramp-rev5-mod      | 3     | 2       | 0        | 1       | 0     |
```

JSON format follows v0.9.0 POA&M list shape for dashboard consumption.

Also add NEW REST endpoint `GET /api/conmon/health` returning same data structure.

NEW module `evidentia_core.conmon.health` (~150 LOC + ~80 LOC tests).
NEW EventAction: `CONMON_HEALTH_REPORT_GENERATED`.

### P1.4 — Integration framework — continuous evidence ingestion plugin

Plugin interface for streaming evidence from external sources into the audit trail. Builds on the v0.8.0 `BaseSaaSCollector` ABC.

Interface:
```python
class ContinuousEvidenceSource(Protocol):
    name: str
    cadence_slug: str
    def poll(self) -> list[Evidence]: ...
    def health_check(self) -> bool: ...
```

Reference implementations (~2):
- `evidentia_integrations.continuous.cloudtrail_aws` — wraps existing v0.7.x AWS collector with a poll method
- `evidentia_integrations.continuous.snyk` — wraps existing v0.7.10 Snyk model-risk collector

Daemon mode: `evidentia conmon watch --continuous --plugin <name>` registers the plugin's poll method on the daemon's cycle.

NEW abstract: `evidentia_core.plugins.ContinuousEvidenceSource` (~80 LOC + ~80 LOC tests).
NEW EventActions: `CONTINUOUS_EVIDENCE_POLL_OK`, `CONTINUOUS_EVIDENCE_POLL_FAILED`, `CONTINUOUS_EVIDENCE_INGESTED`.

## Phase 2 — AI governance foundation (Theme B)

Surfaces an entirely new differentiator dimension. Time-aligned with EU AI Act high-risk obligations (Aug 2026 enforcement).

### P2.1 — EU AI Act catalog enrichment

Existing catalog `eu-ai-act.json` (Tier-D stub) currently has article-level granularity. Enrich to:

- **Sub-article granularity**: each article's paragraphs/points become individual control items (e.g., Article 9.1, 9.2, 9.3 instead of just Article 9).
- **Annex III risk-tier annotations**: each high-risk-AI category (1-8 in Annex III) gets explicit risk_category metadata field; controls mapped to relevant categories.
- **Harmonised-standard cross-references**: as standards stabilize (ISO/IEC standards under EU AI Act Article 40), add `harmonised_standard` field linking to ISO doc numbers.

Stays Tier-A (authoritative public source) — bumps tier classification from D → A.

Catalog growth: ~50 control items → ~150 control items. Updates `data/catalogs/eu-ai-act.json`.

### P2.2 — NIST AI RMF crosswalk authoring

NEW data file: `data/mappings/nist-ai-rmf-to-eu-ai-act.json` and `data/mappings/nist-ai-rmf-to-iso-42001.json`. Same crosswalk pattern as existing `data/mappings/nist-800-53-to-iso-27001.json`.

Human-reviewed mappings from AI RMF subcategories (GOVERN.1.1, MAP.2.3, MEASURE.4.1, MANAGE.3.2 etc.) to:
- EU AI Act articles (5 high-risk + 9 GPAI + 10-15 transparency etc.)
- ISO 42001 Annex A controls (A.6.1 Leadership, A.7.1 Resources, A.8.1 Operational Planning etc.)

~80 mapping entries each = ~160 total.

**Per-mapping confidence flagging** (added per Allen's question-1 sign-off):
- Each mapping entry carries a `confidence` field: `high` (direct semantic match), `medium` (overlapping intent, partial scope), `low` (related domain but ambiguous).
- File header documents the rubric + community-PR pathway for refinement.
- NEW CLI: `evidentia ai-gov crosswalk list --filter confidence=low` surfaces low-confidence mappings for reviewer prioritization.
- Shipping posture: "useful starting point that's better than blank — not FedRAMP-certified mappings." Documented in mapping file headers + the new `docs/ai-gov-crosswalk-methodology.md`.

### P2.3 — AI risk classification module

NEW Python module `evidentia_core.ai_governance.classification` (~300 LOC + ~150 LOC tests).

Classifier logic:
1. **Use-case attribute schema**: `AISystemDescriptor` Pydantic model with fields like `domain` (healthcare/finance/HR/biometric/etc.), `decision_role` (advisory/automated/hybrid), `data_subjects` (consumers/employees/citizens), `output_type` (recommendation/classification/generation), `geographic_scope` (EU/US/global), `purpose` (Annex III categories).
2. **EU AI Act tier classifier**: rule-based classifier maps `AISystemDescriptor` → `EUAIActTier` enum (UNACCEPTABLE / HIGH / LIMITED / MINIMAL). Logic encodes Article 5 prohibitions + Annex III high-risk + Article 50 transparency triggers.
3. **NIST AI RMF risk function classifier**: same descriptor → `NISTAIRMFRiskFunction` enum (GOVERN / MAP / MEASURE / MANAGE) with applicable subcategories.

Output: `AISystemClassification` Pydantic model with both tiers + applicable subcategories + rationale (which rule fired).

### P2.4 — AI system inventory primitives

Data model for registering AI systems. Extends the v0.7.10 model-risk inventory pattern.

- NEW Pydantic models: `AISystemRegistryEntry` (system_id, name, provider, classification, owner, deployment_status, linked_controls, last_assessed_at).
- NEW JSON file-store mirroring v0.7.9 vendor_store + v0.9.0 poam_store: `evidentia_core.ai_governance.registry_store` with atomic-write + UUID-shape ID gate + path-traversal defense + `EVIDENTIA_AI_REGISTRY_DIR` env override.
- NEW EventActions: `AI_SYSTEM_REGISTERED`, `AI_SYSTEM_UPDATED`, `AI_SYSTEM_CLASSIFIED`, `AI_SYSTEM_RETIRED`.

### P2.5 — `evidentia ai-gov classify` CLI verb

NEW Typer subcommand group `evidentia ai-gov` with verbs:

- `evidentia ai-gov classify --descriptor <yaml>` — classify single AI system from YAML descriptor file; outputs `AISystemClassification` JSON
- `evidentia ai-gov register --descriptor <yaml>` — classify + persist to registry
- `evidentia ai-gov list [--tier high|limited|minimal]` — list registered AI systems with filter
- `evidentia ai-gov show <system_id>` — full classification + linked controls

Also NEW REST router `/api/ai-gov/*` with parallel endpoints (5 endpoints; same shape as `/api/poam/*` v0.9.0 + `/api/conmon/*` v0.9.2).

## Phase 4 — Operational hygiene (E1 + E2)

### P4.1 — docker/requirements.txt drift CI gate (E1)

NEW GitHub Action workflow (or step in existing CI): detect drift between `uv.lock` and `docker/requirements.txt` for shared deps.

Implementation:
1. Parse `uv.lock` for top-level deps (skip dev/test groups).
2. Parse `docker/requirements.txt` for the same packages.
3. Compare versions; fail CI if any package's version differs by more than a patch level for security-sensitive packages (urllib3, requests, cryptography, paramiko, aiohttp, httpx).
4. NEW script: `scripts/check_requirements_drift.py` (~120 LOC + ~80 LOC tests).
5. NEW workflow step in `.github/workflows/ci.yml`: `- name: Check requirements drift / run: uv run python scripts/check_requirements_drift.py`.

Closes the structural gap that allowed urllib3 to drift between v0.9.0 → v0.9.2 ships.
