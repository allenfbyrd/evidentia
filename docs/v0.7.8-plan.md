## Theme

**"Cloud data-warehouse evidence collectors + BI output
integrations — Databricks / Snowflake / Tableau / Power BI."**

v0.7.7 added the relational-database evidence layer (5 SQL
adapters). v0.7.8 extends into cloud data warehouses (where
modern regulated-environment data + audit logs increasingly
live) and adds the first **output integrations to BI tools**
— positioning Evidentia as the OSS evidence-feed beneath
enterprise dashboards.

Each cloud-DW collector follows the same enterprise-grade
pattern as v0.7.7 SQL collectors: typed exception hierarchy,
`@with_retry`, ECS-structured logging, explicit `BLIND_SPOTS`,
read-only principal verification, air-gap mode validation.

The Tableau + Power BI integrations are the first **output
integrations** since Jira (v0.5.0) — they push compliance
artifacts (gap reports + risk register data) to enterprise
BI surfaces that risk officers + audit committees + boards
already consume.

This is a substantial release. Realistic ship target: 6-8 weeks
after v0.7.7.

## P0 — Cloud data-warehouse collectors

### P0.1 — Databricks (`evidentia-collectors[databricks]`)

| What | Source | NIST mappings |
|---|---|---|
| Workspace audit logs | Unity Catalog `system.access.audit` (workspace audit) | AU-2, AU-3 |
| Table + column lineage | `system.access.column_lineage`, `system.access.table_lineage` | SI-7 (info integrity); SR 11-7 model lineage (financial overlay) |
| Cluster compliance | `system.compute.clusters` (runtime version, libraries, init scripts) | CM-2, CM-3, SI-2 |
| Workspace network policies | Account API: workspaces, network configurations | SC-7 |
| Service Principal usage | `system.access.audit` filtered to SP actions | AC-2, AC-3 |
| Secret scopes | Workspaces API `2.0/secrets/scopes/list` | SC-12 |
| Personal Access Token (PAT) inventory | Tokens API `2.0/token-management/tokens` | AC-2, IA-5 |

CLI: `evidentia collect databricks --workspace-url https://... --token-env DATABRICKS_TOKEN`

Auth: PAT via env var, or OAuth M2M client credentials. Service
Principal recommended for production.

### P0.2 — Snowflake (`evidentia-collectors[snowflake]`)

| What | Source | NIST mappings |
|---|---|---|
| Login history | `account_usage.LOGIN_HISTORY` (90-day window standard) | AC-7, AU-2 |
| User + grant inventory | `account_usage.USERS`, `account_usage.GRANTS_TO_USERS`, `account_usage.GRANTS_TO_ROLES` | AC-2, AC-3, AC-6 |
| Access history (data lineage) | `account_usage.ACCESS_HISTORY`, `account_usage.OBJECT_DEPENDENCIES` | SI-7; SR 11-7 model lineage |
| Network policies | `INFORMATION_SCHEMA.NETWORK_POLICIES`, `account_usage.NETWORK_POLICIES_HISTORY` | SC-7 |
| Masking + row-access policies | `INFORMATION_SCHEMA.MASKING_POLICIES`, `INFORMATION_SCHEMA.ROW_ACCESS_POLICIES` | AC-3, SC-28 |
| Key rotation status | `INFORMATION_SCHEMA.KEY_DETAILS` (per-account encryption key rotation) | SC-12 |
| MFA enforcement | `account_usage.USERS` filtered to `HAS_MFA = TRUE`; aggregate by role | IA-2 |
| Failed-login spike detection | derived from `LOGIN_HISTORY` over a sliding 24h window | AC-7, IR-4 |

CLI: `evidentia collect snowflake --account ... --user ... --password-env SNOWFLAKE_PWD`

Auth: password via env var, or OAuth, or key-pair (recommended for
production — Snowflake is deprecating password auth).

### Why both Databricks AND Snowflake (not one or the other)

The two platforms have non-overlapping strengths in regulated
environments:
- Databricks dominates ML/AI workloads with model lineage as
  first-class data → useful for SR 11-7 model-risk evidence
  (v0.7.9 overlay).
- Snowflake dominates data-warehouse / SOX-compliant data
  controls → useful for SOX 302/404 IT-general-controls
  evidence.

Most regulated enterprises run both. v0.7.8 ships both adapters
with the same surface contract so a single Evidentia run can
collect from both in one OSCAL Assessment Results document.

## P1 — BI output integrations

### P1.1 — Tableau (`evidentia-integrations[tableau]`)

**What**: publish gap reports + risk register data to Tableau
Server / Tableau Cloud as a refreshable extract.

**How**: Tableau REST API → publish a `.hyper` extract (Tableau's
columnar format) of:
- Gap inventory (one row per gap with framework, control_id,
  severity, status, owner, remediation cost estimate)
- Risk statements (NIST SP 800-30 fields: threat, vulnerability,
  likelihood, impact, narrative)
- Evidence collection runs (CollectionContext metadata for audit
  trail dashboards)

CLI: `evidentia integrations tableau publish --gaps report.json --site-id <site> --project-id <project>`

REST: `POST /api/integrations/tableau/publish`

Auth: Tableau Personal Access Token via `TABLEAU_PAT` env var, or
JWT-based connected-app auth (recommended for production).

**Tableau-specific feature**: ships a starter Tableau workbook
template at `examples/tableau/evidentia-compliance-dashboard.twb`
that consumes the published extract — risk officers get an
end-to-end working dashboard in one ship.

### P1.2 — Power BI (`evidentia-integrations[powerbi]`)

**What**: same as Tableau — push compliance datasets to a Power BI
workspace.

**How**: Power BI Push Datasets API → push gap inventory + risk
statements + collection-run metadata as Power BI streaming
datasets. Optional Power BI Embedded support documented (not in
the OSS scope; reference impl).

CLI: `evidentia integrations powerbi publish --gaps report.json --workspace-id <ws> --dataset-id <ds>`

REST: `POST /api/integrations/powerbi/publish`

Auth: Azure AD Service Principal with appropriate Power BI
workspace permissions. Auth flow uses MSAL Python.

**Power BI-specific feature**: ships a starter Power BI report
template at `examples/powerbi/evidentia-compliance.pbit` (Power BI
template format, no embedded data) that consumes the published
dataset.

### Why both Tableau AND Power BI

- Tableau dominates the legacy enterprise + financial-services
  market.
- Power BI dominates Microsoft-shop deployments + post-2020 net
  new dashboards.
- Most regulated enterprises run both. Same OSS-substrate
  positioning.

## Architectural notes

### Test infrastructure for cloud DWs

Cloud DWs are harder to integration-test than relational DBs
because they don't ship as Docker images. Approach:
- **Unit tests**: mock the platform SDK clients (databricks-sdk,
  snowflake-connector-python) via `unittest.mock` + recorded
  fixture responses
- **Integration tests** (gated `@pytest.mark.integration` +
  `@pytest.mark.cloud`) — run only when env vars
  `EVIDENTIA_DATABRICKS_TEST_WORKSPACE_URL`, `EVIDENTIA_SNOWFLAKE_TEST_ACCOUNT`,
  etc. are set. CI runs these in a manual job triggered after
  the v0.7.8 RC against Allen's personal sandbox accounts.
- **Recorded-response fixtures**: capture once with `pytest-recording`
  (vcr.py-based); replay deterministically thereafter.

### BI integration test approach

Tableau Server + Tableau Cloud + Power BI all support test/dev
tenants. Recommended:
- Set up a personal Tableau Cloud test site (free 14-day; can
  extend with developer license)
- Set up a Power BI Embedded development workspace
- Both tested only at RC time, not in CI (no free CI-friendly
  test instances exist)

### `BLIND_SPOTS` disclosure for cloud DWs

- `EVIDENTIA-DATABRICKS-UC-NOT-ENABLED` — workspaces without
  Unity Catalog have no `system.access.audit`; collector
  emits a finding pointing operators at the upgrade path
  rather than failing silently
- `EVIDENTIA-SNOWFLAKE-90D-RETENTION` — `account_usage` views
  cap at 365 days (most at 90); collector documents the
  effective look-back window per query
- `EVIDENTIA-DATABRICKS-CLUSTER-INIT-SCRIPT-CONTENT` — collector
  reports init-script presence + filename but cannot
  inspect script contents (security boundary); operator
  must review separately
- `EVIDENTIA-SNOWFLAKE-INFORMATION-SCHEMA-EDITION-LIMIT` — some
  views (e.g., `KEY_DETAILS`) require Enterprise Edition or
  higher; collector emits a finding when query fails due to
  edition limit
