# Evidentia v0.9.9 plan — Supply-chain hygiene + pre-push gate fidelity

> **Status**: SHIPPED 2026-05-21 (drafted + executed same-day).
> **Theme**: *Supply-chain hygiene + pre-push gate fidelity* — closes
> the `paramiko` CVE carried forward from v0.9.8, adds an
> `osv-scanner --sbom` pre-push gate, and clears the entire Dependabot
> PR queue.
> **Pacing**: Single focused patch session. NOT the "pre-1.0.0" ship.
> **Canonical location**: `docs/v0.9.9-plan.md`.

## Theme rationale

v0.9.8 (2026-05-21) shipped the v0.9.7 deferral-closure + v1.0-prep
integration wiring and bucketed three items forward to v0.9.9:
`paramiko` CVE-2026-44405, an `osv-scanner --sbom` pre-push gate, and
the federal-SI walk-through. Pre-cycle investigation reshaped the
scope:

- **paramiko is a dev-only transitive dependency.** No Evidentia code
  imports `paramiko`; there is no SSH collector. It enters only via
  `compliance-trestle` (a dev dependency, for OSCAL round-trip
  tests). CVE-2026-44405 was never on a reachable path — but a fix
  now exists, so the cycle closes it for hygiene.
- **The fix was already pre-verified.** `compliance-trestle` 4.0.3
  pins `paramiko==5.0.0` and sits inside the existing `>=4.0,<5.0`
  constraint; Dependabot's `python-dev` group PR carried it CI-green.
- **The gate-fidelity gap is real.** `osv-scanner` ran nowhere — only
  as a manual post-tag check. A disputed `pyjwt` CVE surfaced
  post-tag in v0.9.8 because the pre-push gate's Row 14 read
  Dependabot alerts, which suppress disputed CVEs.

Per the v1.0 master-plan resequencing (2026-05-21), v0.9.9 is a
routine patch — the 0.9.x line now iterates as many times as needed
toward a solid product, and the federal-SI walk-through is deferred
indefinitely (before v1.0.0, after the operator self-test +
demo/pitch phase). See `docs/ROADMAP.md`.

## Phase 1 — Dependabot queue clearance

- **1.1** Merge the five current-config grouped version-update PRs
  (`python-dev`, `npm-runtime`, `npm-dev`, `github-actions`, the
  Docker base-image digest) — all CI-green. The `python-dev` group
  carries `compliance-trestle` 4.0.2 → 4.0.3.
- **1.2** Close three orphaned PRs that targeted only
  `docker/requirements.txt` via a `pip`/`uv-docker` Dependabot
  ecosystem no longer present in `.github/dependabot.yml`. That file
  is regenerated from `uv.lock` at release time (`release.yml` G4
  Path 2); the orphaned PRs were superseded.
- **1.3** Audit `.github/dependabot.yml` — `uv` (all 7 pyproject via
  `uv.lock`), `npm`, `github-actions`, `docker` (base image). No
  coverage gap: `docker/requirements.txt` is release-regenerated and
  correctly needs no separate `pip` watcher.
- **1.4** Prune the stale `dependabot/*` remote-tracking refs.

## Phase 2 — `osv-scanner --sbom` pre-push gate

- NEW `scripts/run_osv_scan.py` — one shared entry point: generates
  the CycloneDX SBOM (the exact command `release.yml` uses) and scans
  it with `osv-scanner`, honouring `osv-scanner.toml`. A multi-form
  invocation lets the gate survive osv-scanner CLI changes.
- NEW `osv-scanner.toml` — allowlist; one entry (`pyjwt`
  PYSEC-2025-183, disputed) with a reason + an `ignoreUntil` date.
- NEW `osv-scan` job in `.github/workflows/test.yml` — installs a
  pinned, SHA256-verified `osv-scanner` v2.3.8 and runs the shared
  script against the same dependency closure as the release
  checklist.
- `docs/release-checklist.md` Step 5 gains the same
  `python scripts/run_osv_scan.py` invocation. CI and the documented
  gate run one shared check — the gate-fidelity end state called for
  by the v0.9.8 lesson.
