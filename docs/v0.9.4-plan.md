# Evidentia v0.9.4 plan — daemon hardening + operator polish + walk-through

> **Status**: PLANNED (drafted 2026-05-17 during v0.9.3 pre-tag review)
> **Theme**: *daemon hardening + operator polish + walk-through scenarios*
> **Pacing target**: Conservative ~2-3 weeks (vs v0.9.3's aggressive
> ~3 weeks). v0.9.3 was the largest minor of the v0.9.x line; v0.9.4
> is the consolidation pass that closes operator-feedback items + the
> walk-through that was deferred from v0.9.0/v0.9.1.
> **Canonical location**: `docs/v0.9.4-plan.md` (committed in-repo at
> cycle-open per the institutional-knowledge pattern)

---

## Theme rationale

v0.9.3 shipped two major new feature surfaces (CONMON daemon + AI
governance) in a single aggressive cycle. The pre-release review
surfaced 28 findings (0 CRITICAL, 1 HIGH, 10 MEDIUM, 11 LOW, 6 INFO);
8 of 10 MEDIUMs fixed in cycle. v0.9.4's theme is consolidation:

1. **Daemon hardening** — close the 2 deferred MEDIUMs and the
   design-laden HIGH (F-V93-Q3 file-locking; F-V93-S2 webhook SSRF)
2. **Operator polish** — clear the LOW-bucket polish items that
   surface in operator deployment scenarios
3. **Walk-through scenarios** — execute the federal-SI walk-through
   that was deferred from v0.9.0 and remained outside v0.9.3 scope

This is the OPPOSITE pattern from v0.9.3 (single-session compression).
v0.9.4 deliberately spreads across multiple sessions with operator-
feedback collection between phases — matching the "walk-through
opener" model that was reserved at v0.8.7 cycle-close.

## Phase 1 — Daemon hardening (closes deferred review items)

### P1.1 — File-locking helper for state file (closes F-V93-Q3 HIGH)

The v0.9.3 review documented the single-writer contract for
`mark_completed` and `AlertDeduper.mark_dispatched` because both do
non-atomic read-modify-write. v0.9.4 adds an optional helper for
operators wanting multi-writer semantics.

- New module `evidentia_core.security.file_lock` exposing:
  - `FileLock(path)` context manager (POSIX `fcntl.flock` +
    Windows `msvcrt.locking`)
  - `with_locked_state(path, mode="json"|"yaml")` higher-level
    helper for read-modify-write cycles
- Wire `mark_completed` + `AlertDeduper.mark_dispatched` behind an
  opt-in `--state-lock` flag (default off — preserves
  v0.9.3 backward-compat single-writer perf path)
- Test coverage: parallel `pytest-xdist` test exercising 4 concurrent
  writers + asserting no clobbering
- **Risk acceptance owner**: per v0.9.3 review: Allen, with re-check
  date 2026-08-01 (3 months after v0.9.3 ship)

### P1.2 — Webhook SSRF mitigation (closes F-V93-S2 MEDIUM)

The v0.9.3 review deferred this MEDIUM because rushing a 1-day
mitigation risks breaking legitimate internal-webhook deployments
(operators on RFC1918 internal HTTPS webhooks need a sanctioned
path).

- Default: reject `http://` schemes; reject loopback / RFC1918 /
  link-local / RFC6890 reserved ranges
- Opt-in flags: `--webhook-allow-plaintext` (only http://) and
  `--webhook-allow-private-network` (only RFC1918/loopback)
- Documentation: explicit threat-model note in
  `docs/conmon-daemon-deployment.md` calling out the SSRF threat +
  cloud metadata service exfiltration vector
- Test coverage: 4 new unit tests covering each rejected scheme +
  the opt-in passthrough behavior

### P1.3 — Rate-limit + idempotency on AI gov register (closes F-V93-S10 LOW)

- FastAPI middleware integration with a stdlib-only rate-limiter
  (per-client-IP token bucket, no external dep)
- Optional `X-Idempotency-Key` header support on
  `POST /api/ai-gov/register`: identical key + identical body returns
  the prior `system_id` instead of creating a duplicate

### P1.4 — Polish items batch (closes F-V93-Q11, Q12, Q14, S9)

Single commit covering:
- F-V93-Q11: Use `evidentia_core.__version__` for webhook User-Agent
  (drop hardcoded `v0.9.3` string)
- F-V93-Q12: Document Windows signal-latency in
  `docs/conmon-daemon-deployment.md`
- F-V93-Q14: Narrow `_load_descriptor` `except Exception` to
  `(ValidationError, ValueError)` in `cli/ai_gov.py`
- F-V93-S9: Document path-disclosure in audit-event payloads in
  `docs/log-schema.md`; recommend operators scrub before SIEM

## Phase 2 — Operator polish

### P2.1 — Daemon health endpoint

- `GET /api/conmon/daemon-status` — returns:
  - PID file location (from configurable path)
  - last poll timestamp
  - last poll outcome (success | failed)
  - tracked-cadence count
  - poll-interval in effect
- Pairs with the CONMON_DAEMON_POLL_FAILED action shipped in v0.9.3

### P2.2 — Dedup state inspection CLI

`evidentia conmon dedup-list [--slug X]` — operator-facing read of
the alert-dedup state file. Shows per-(slug, state) last-dispatched
timestamps + suppression-window status. Helpful for "why didn't I get
an alert?" debugging.

### P2.3 — AI gov update + retire CLI verbs

The v0.9.3 ship reserved `AI_SYSTEM_UPDATED` and `AI_SYSTEM_RETIRED`
EventActions but didn't wire the corresponding CLI verbs (only the
REST DELETE wired RETIRED, and only via implicit semantic mapping
per Step 5.A fix).

- `evidentia ai-gov update <system-id> [--owner X] [--deployment-status Y]`
- `evidentia ai-gov retire <system-id>` (sets `deployment_status=retired`
  WITHOUT deleting the entry; preserves historical record for audit)

### P4.4 — Test stability: fix flaky `TestJiraStatus::test_returns_auth_error_when_credentials_reject` (NEW, surfaced post-v0.9.3-tag)

The Jira integration auth-error assertion test
(`tests/integration/test_api/test_integrations.py`, originally added
in v0.5.0) **flakes intermittently on ubuntu-latest only**. Observed
on the v0.9.3 merge-commit CI run (commit `a5a6c02`, 2026-05-17
05:16:43 UTC, run `25982167696`):

- macos-latest pytest job: PASS
- windows-latest pytest job: PASS
- ubuntu-latest pytest job: 1 FAILED with
  `assert '401' not in r.text` —  response body contained "401"
  somewhere; on the other 2 OS the response had a generic sanitized
  error message
- Local re-run on Windows: PASS
- Same code passed CI 2 minutes earlier on the PR branch HEAD
  (`a2eb525`) on all 3 OS

Classic test-isolation/ordering race signature: likely a sibling
test sets up a Jira httpx mock that returns an actual `401` body
without sanitization, and the mock's state leaks into
`test_returns_auth_error_when_credentials_reject` when collected in
a specific order on ubuntu-latest only. Not a v0.9.3 regression and
not a real auth-error-leak bug in the product (the sanitization code
DOES work — proven by macos/windows/local passing on the same SHA).

**Resolution plan**:

1. Add `pytest-randomly` to `[dev]` deps + remove `pytest-xdist` if
   present (parallel collection can interact with the ordering issue)
2. Run the test suite locally with `--randomly-seed=12345` (or a
   parametrized seed sweep) to reproduce the ordering that triggers
   the leak
3. Identify the sibling test whose mock/fixture is leaking
4. Tighten the fixture scope (`function` not `class`/`module`) or add
   explicit teardown (`monkeypatch.undo()` / `httpx_mock.reset()`)
5. Add a `pytest.mark.flaky(reruns=2)` decorator as a safety net if
   the root cause turns out to be transient (e.g., race against the
   in-process FastAPI TestClient)

This is a v0.9.4 P4.4 hygiene item — does NOT block v0.9.3 ship
(already on PyPI + GHCR + tagged + Step 7 verified clean). Flagging
here so the v0.9.4 cycle catches it before the next minor.

### P4.5 — Add `workflow_dispatch` trigger to `test.yml` (NEW)

Discovered during the v0.9.3 post-tag flaky-test investigation:
`gh workflow run test.yml --ref main` returns
`HTTP 422: Workflow does not have 'workflow_dispatch' trigger`. Means
operators can't manually re-trigger the test workflow to validate
flakiness without pushing an empty commit or opening a no-op PR.

Trivial fix — add to `.github/workflows/test.yml`:

```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  workflow_dispatch:   # NEW v0.9.4 P4.5
```

Same change recommended for `release.yml` if it isn't already there
(it already supports `workflow_dispatch` per the v0.7.x bump-type
picker pattern; v0.9.4 verifies).

### P4.6 — Codecov + `gh secret set` token-replacement gotcha (NEW)

Discovered during the v0.9.3 ship: my Codecov token-replacement
recipe surfaced `--body-file` which doesn't exist on `gh secret set`
(only `-b/--body string`, `-f/--env-file file`, or stdin). Update the
`docs/release-checklist.md` token-rotation guidance + the
`security-review-v0.9.3.md` § "F-V93-Codecov-Stale-Org-Activation"
fix recipe to use one of:

- `gh secret set -R <repo> -f <dotenv-formatted-file>` (file has
  `KEY=value` lines)
- `Get-Content <raw-token-file> | gh secret set CODECOV_TOKEN -R <repo>`
  (PowerShell pipe; file has bare value only)

The dotenv `-f` form is cleaner for multi-secret rotations; the stdin
pipe is cleaner for single-secret rotations.
