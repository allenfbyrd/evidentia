## What we got right

**Audit-event registry as canonical schema.** The
`evidentia_core.audit.EventAction` enum became the de-facto
public schema for every observable event. Adding events is
additive; auditors get a stable vocabulary; v0.8.x extensions
(DFAH, PRT, MCP, CIMD) all flowed through the same enum +
ECS-aware logger pattern.

**Structured logging from day one.** ECS field shape +
`evidentia.*` extension namespace + `extra={"ecs_record": ...}`
emit pattern made downstream observability tooling (Loki,
Splunk, Datadog) drop-in compatible. Zero "log format breaking
change" support tickets across the cycle.

**PEP 740 + Sigstore + SLSA L3 from v0.7.0.** Supply-chain
posture was strong out of the gate. Shipping the verification
recipe (`pypi-attestations verify pypi`, `cosign verify`,
`osv-scanner --sbom`) in every release narrative made it easy
for downstream operators to do their own posture checks.

**Retention + WORM as first-class library surface.** The
v0.7.11 → v0.7.12 retention work landed enough abstraction
that S3 + Azure + GCS impls all conformed to one ABC; GDPR
Article 17 purge flow snapped in cleanly.

**Pin-trap fix as structural posture.** v0.7.12's
`scripts/bump_version.py` atomic inter-package pin update
fixed a recurring foot-gun that had haunted v0.7.6 → v0.7.11
(13 manual hot-fix tags caused by stale inter-package pins
post-publish). 5 consecutive releases shipped clean post-fix.

**release.yml CHANGELOG auto-population.** v0.7.13's
`extract_changelog_block.py` + release.yml integration meant
every subsequent release body was substantive on first publish
— no more "stub release body, manual edit needed" loop.
4 consecutive releases auto-populated cleanly through cycle
close.

## What slipped

**DFAH determinism harness landed in v0.8.0, not v0.7.x.**
Originally scoped for v0.7.x; the AI-quality work needed more
time + landed as the v0.8.0 P0.1 flagship. Honest deferral
documented across multiple cycle plans.

**MCP CIMD richness — 5-cycle deferral.** Reserved in v0.8.0
§5.3, deferred to v0.8.1, v0.8.2, v0.8.3, v0.8.4 — landed in
v0.8.5 P4 only after explicit "implement now" directive. Per
the §24.6 R6 framing ("infra primitives best explored against
real operator deployments vs guessed at"), the deferral
pattern was rational; we shipped only when the broader CIMD
infrastructure (registry + scope + audit-trail) had a clear
shape. Per-tool scope enforcement at MCP-protocol level
itself shipped in v0.8.6 P1.

**Codecov 0% from v0.7.10 → v0.7.14.** A coverage-upload
configuration regression silently broke for ~5 releases.
Diagnosed in v0.7.14 P2.1 (`source_pkgs` config fix for
multi-package monorepo); 82.14% coverage restored.

**G4 Path 1 cross-platform reproducibility.** v0.8.3 attempted
SOURCE_DATE_EPOCH-driven `uv build` for byte-identical wheels
across Windows + Linux. First-fire revealed `uv build` is NOT
byte-identical between hosts even with same SOURCE_DATE_EPOCH
(file-ordering / timestamp-precision drift). Hot-fix v0.8.3.1
reverted; v0.8.4 G4 Path 2 (post-PyPI regeneration in
release.yml) closed it structurally. The ATTEMPT was correct
per industry guidance; the EMPIRICAL reality required a
different path.

**Frontend modernization deferred across cycles.** Tailwind 3
→ 4 + TypeScript 5 → 6 + ESLint 9 → 10 sat in PRs #18 + #21
through v0.7.13; v0.7.14 + v0.7.15 absorbed them. Cycle had
to budget for major-bump time-boxing (R1 risk in v0.7.14
plan).

## Carries into v0.8.x

**Audit + observability primitives.** Every v0.8.x release
extends the same ECS schema + EventAction enum + ECS-aware
logger; v0.8.6 P1 alone added 2 new EventActions
(`AI_MCP_TOOL_AUTHORIZED` + `AI_MCP_TOOL_DENIED`).

**Supply-chain posture as cycle-default.** PEP 740 + Sigstore
+ SLSA L3 + osv-scanner + Trivy + Scorecard + cosign + SBOM
all run on every v0.8.x tag without per-release configuration
work. The G4 Path 2 closure in v0.8.4 made the recurring
PinnedDependencies false-positive cycle structurally extinct.

**Pin-trap fix carries forward.** 12 consecutive ships
through v0.8.5 with first-attempt pin-trap fresh-venv install
validation. Inter-package release coordination is a solved
problem.

**Retention library surface.** v0.8.x cycles extend retention
metadata to new artifacts (eval results in v0.8.2 P3.2;
faithfulness reports in v0.8.4); the v0.7.11/12 ABC pattern
absorbs them cleanly.

**Standing-rule sweep + pre-commit hook.** v0.7.15 + v0.7.16
introduced the standing-rule keyword sweep at file-content +
commit-msg stages. Every v0.8.x commit + push runs through it
before staging.

**Release.yml CHANGELOG auto-populate.** 11 consecutive
releases through v0.8.5 with auto-populated bodies ≥ 1500
bytes on first publish.

## Pre-release-review v4 cadence

The v4 SKILL.md pre-release-review skill stabilized through
the v0.7.x cycle:

- v0.7.0 → v0.7.5: skill was being prototyped; some cycles
  shipped without it.
- v0.7.6 → v0.7.10: skill was running per-release but with
  variable rigor.
- v0.7.11 → v0.7.16: skill stabilized at v4 with mandatory
  `/security-review` at Steps 3 + 4 + 6.C, mandatory
  `/code-review` auto-fire on 4 triggers, 5 canonical Pre-tag
  deliverables, 16-row pre-push gate.

By v0.7.16 the cycle close had shipped 6 consecutive
PROCEED-CLEAN; v0.8.x continued the streak (v0.8.0 → v0.8.5 =
6 more). The skill itself is a v0.7.x artifact carried
forward — every v0.8.x cycle ships through it.

## What this cycle taught us

**Aggressive cadence pairs well with strict per-cycle
acceptance.** 18 releases in 12 days only worked because
every release had to clear the same 16-row pre-push gate +
0-unfixed-CRITICAL/HIGH posture. Without that rigor, the
cadence would have shipped regressions; with it, the cadence
was a forcing-function for clean carry-forwards.

**Plan files are the durability layer.** The internal
`~/.claude/plans/please-make-a-comprehensive-bright-hopcroft.md`
captured per-cycle Q&A lock-ins that survived session
boundaries — every cycle opened fresh from the prior cycle's
explicit decisions. The pattern works.

**Retrospective docs land late but pay forward.** This doc is
written 2 days into v0.8.6 (cycle that opened with explicit
"v0.7.x retrospective" + "v1.0 transition draft" additions);
the v1.0 transition narrative draft (`docs/v1.0-transition.md`)
references this doc as input. The pair work better together
than either alone.
