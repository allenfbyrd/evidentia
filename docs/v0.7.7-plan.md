## Theme

**"SQL-family evidence collectors — Postgres / MySQL / SQLite /
MS SQL Server / Oracle."**

Evidentia v0.5.0 shipped collectors for AWS, GitHub, and Jira
(integration). The relational-database tier — where compliance-
relevant evidence about user privileges, audit-log retention,
encryption-at-rest config, and schema change control actually
lives in regulated environments — was deferred. v0.7.7 closes
that gap.

Each adapter is **read-only by design**, surfaces a
`SecurityFinding` per row collected, maps each finding to the
relevant NIST 800-53 controls (AC-2 / AC-3 / AC-6 / AU-2 / AU-3 /
SC-12 / SC-28), and follows the v0.7.0 enterprise-grade collector
pattern: typed exception hierarchy, `@with_retry` for transient
failures, ECS-structured logging via `evidentia_core.audit.logger`,
explicit `BLIND_SPOTS` disclosure list per adapter, audit-trail
metadata via `CollectionContext` on every output.

**Industry context**: every regulated financial institution runs
risk-and-compliance programs that depend on database-resident
evidence — model input/output lineage (SR 11-7), user-privilege
attestation (SOX 302/404 IT-general controls), audit-log
retention (FFIEC IT Examination Handbook §Audit), encryption
posture (GLBA Safeguards). v0.7.7 SQL collectors map directly to
those use cases without naming any specific industry overlay
(the financial-sector overlay catalogs land in v0.7.9).

This is a substantial release. Realistic ship target: 6-8 weeks
after v0.7.6.

## P0 — SQL family collectors

New workspace extra: `evidentia-collectors[sql]`. Each adapter is
its own optional sub-extra so users only install the connectors
they need.

### P0.1 — PostgreSQL (`[sql-postgres]`)

| What | Source | NIST mappings |
|---|---|---|
| User + role inventory | `pg_roles`, `pg_authid`, `pg_auth_members` | AC-2 |
| Privilege grants | `INFORMATION_SCHEMA.TABLE_PRIVILEGES`, `INFORMATION_SCHEMA.ROUTINE_PRIVILEGES`, `pg_catalog.pg_class.relacl` | AC-3, AC-6 |
| Audit log status | `pg_settings` for `log_*` config; pgaudit extension installed/enabled if present | AU-2, AU-3 |
| Encryption at rest | `pg_settings.ssl_*`, FILESYSTEM-LEVEL note (Postgres relies on filesystem TDE) | SC-28 (with documented BLIND_SPOT) |
| Crypto config | `pg_settings.password_encryption` (must be `scram-sha-256`), TLS settings | SC-12 |
| Connection limits | `pg_settings.max_connections`, `pg_hba.conf` if readable | AC-3 |

CLI: `evidentia collect sql --adapter postgres --connection-uri postgres://user@host/db?sslmode=require`

REST: `POST /api/collectors/sql/postgres/collect`

Connection via `DATABASE_URL`-style env var; never via CLI flag
(per the project's existing credential-handling pattern). The
adapter requires a read-only DB principal; if granted write
privilege, errors out loudly with a NIST-AC-6-violation finding.

### P0.2 — MySQL / MariaDB (`[sql-mysql]`)

| What | Source | NIST mappings |
|---|---|---|
| User + role inventory | `mysql.user`, `mysql.role_edges` (8.0+) | AC-2 |
| Privilege grants | `INFORMATION_SCHEMA.USER_PRIVILEGES`, `INFORMATION_SCHEMA.SCHEMA_PRIVILEGES`, `INFORMATION_SCHEMA.TABLE_PRIVILEGES` | AC-3, AC-6 |
| Audit log status | `general_log` setting; `audit_log_*` if MariaDB Enterprise / Percona Audit | AU-2, AU-3 |
| TLS/encryption | `SHOW VARIABLES LIKE 'have_ssl'`, `ssl_*`, `require_secure_transport` | SC-12 |
| Encryption at rest | InnoDB tablespace encryption status, keyring plugin | SC-28 |
| Connection limits | `max_connections`, `max_user_connections` | AC-3 |

CLI: `evidentia collect sql --adapter mysql --host ... --user ... --password-env MYSQL_PWD`

### P0.3 — SQLite (`[sql-sqlite]`)

Smaller surface — SQLite has no built-in user system; collector
focuses on file-level + extension-level evidence:

| What | Source | NIST mappings |
|---|---|---|
| Journal mode | `PRAGMA journal_mode`, `PRAGMA synchronous` | SC-28 (durability evidence) |
| Encryption | SEE / SQLCipher / WxSQLite3 detection via `PRAGMA cipher_version` | SC-28 |
| Schema integrity | `PRAGMA integrity_check`, `PRAGMA foreign_key_check` | SI-7 (information integrity) |
| File ACLs | OS-level file permissions of the database file (UNIX `stat` / Windows ACL) | AC-3 (limited; documented BLIND_SPOT for distributed deployments) |

CLI: `evidentia collect sql --adapter sqlite --database /path/to/db.sqlite`

Lower priority within P0 since SQLite is uncommon in regulated
production environments — but useful for the broader OSS user
base + for completeness.

### P0.4 — MS SQL Server (`[sql-mssql]`)

| What | Source | NIST mappings |
|---|---|---|
| User + role inventory | `sys.server_principals`, `sys.database_principals`, `sys.server_role_members`, `sys.database_role_members` | AC-2 |
| Privilege grants | `sys.server_permissions`, `sys.database_permissions`, `sys.fn_my_permissions` | AC-3, AC-6 |
| Audit log status | `sys.fn_get_audit_file()`, `sys.server_audits`, `sys.server_audit_specifications`, `sys.database_audit_specifications` | AU-2, AU-3 |
| TDE status | `sys.dm_database_encryption_keys`, `sys.databases.is_encrypted` | SC-28 |
| Always Encrypted columns | `sys.column_encryption_keys`, `sys.column_master_keys` | SC-28 |
| TLS connection | `sys.dm_exec_connections.encrypt_option` | SC-12 |
| Audit policy | `sys.policies` (Policy-Based Management) | CM-3 |

CLI: `evidentia collect sql --adapter mssql --server ... --database ... --auth-mode {windows,sql,azure-ad} --password-env MSSQL_PWD`

### P0.5 — Oracle Database (`[sql-oracle]`)

| What | Source | NIST mappings |
|---|---|---|
| User + role inventory | `DBA_USERS`, `DBA_ROLES`, `DBA_ROLE_PRIVS`, `DBA_SYS_PRIVS`, `DBA_TAB_PRIVS` | AC-2, AC-3, AC-6 |
| Profile + password policy | `DBA_PROFILES` (PASSWORD_LIFE_TIME, FAILED_LOGIN_ATTEMPTS, etc.) | IA-5 |
| Audit log status | `DBA_AUDIT_TRAIL` (legacy unified pre-12c) or `UNIFIED_AUDIT_TRAIL` (12c+) | AU-2, AU-3 |
| TDE status | `V$ENCRYPTION_WALLET`, `DBA_TABLESPACES.ENCRYPTED`, `DBA_TDE_TABLE_ENCRYPTION` | SC-28 |
| Network encryption | `V$SESSION_CONNECT_INFO.NETWORK_SERVICE_BANNER` | SC-12 |
| Database Vault status | `DBA_DV_REALM`, `DBA_DV_RULE_SET` if Database Vault licensed | AC-3, AC-6 |

CLI: `evidentia collect sql --adapter oracle --connection-string "..." --password-env ORACLE_PWD`

Uses `oracledb` (Oracle's modern thin driver — pure Python, no
Instant Client required for basic collection; thick mode supported
for advanced features).

## Architectural notes

### Test infrastructure

- **Unit tests**: pytest fixtures + Docker-pytest-fixtures for
  Postgres / MySQL / MSSQL test instances; SQLite uses in-process
  `:memory:` databases; Oracle uses the Oracle DB Free image
  (~3 GB but free + suitable for CI when needed)
- **Integration tests** in `tests/integration/test_sql/` —
  marked with `@pytest.mark.integration` and skipped on the
  default test run; CI's `test.yml` runs them in a separate job
  with the Docker images warm
- **Fixture data**: synthetic schemas representing typical
  compliance-relevant table shapes (audit_log, users, roles,
  user_role_assignments, sensitive_data) so each adapter has
  consistent ground-truth findings

### Connection-string handling

Per the existing v0.7.0 collector pattern + the secret-handling
protocol in `~/.claude/CLAUDE.md`:
- Connection passwords NEVER pass via CLI flags
- Required env-var format: `EVIDENTIA_<ADAPTER>_PASSWORD` (e.g.,
  `EVIDENTIA_POSTGRES_PASSWORD`)
- Connection URIs without passwords ARE allowed via CLI
- `evidentia doctor --check-credentials` validates the env-var
  set is complete before invoking the collector
- Air-gap mode (`--offline`): SQL collectors validate that the
  connection target is RFC-1918 / loopback / explicitly-allowed
  before opening the connection. Documented in
  `docs/air-gapped.md` updates

### Read-only principal verification

Each adapter on first connect runs a quick "can I write?" probe
(e.g., for Postgres: `SELECT current_setting('default_transaction_read_only')`
+ attempt a `CREATE TEMP TABLE` and roll back). If write privilege
is detected, the collector emits a finding tagged `EVIDENTIA-WRITE-PRIV-DETECTED`
mapped to `AC-6` (least privilege violation) and continues read-only
collection. Production deployments should grant the read-only
principal explicitly.

### `BLIND_SPOTS` disclosure (v0.7.0 pattern)

Each adapter ships a documented `BLIND_SPOTS` list as a Python
constant + as a back-matter resource on each collected
`AssessmentResults` document. Examples:
- `EVIDENTIA-SQLITE-FILE-ACL-MULTI-HOST` — SQLite file ACLs are
  meaningful only on single-host deployments; distributed-fs
  scenarios produce inaccurate findings.
- `EVIDENTIA-MYSQL-AUDIT-PLUGIN-COMMUNITY` — MySQL Community
  Edition has no built-in audit log; only general_log is
  available, which has known performance + retention limitations.
- `EVIDENTIA-ORACLE-LICENSE-FEATURE` — Oracle Database Vault and
  Advanced Security Option are separately licensed; collector
  reports their presence/absence but cannot evaluate their
  configuration without those licenses.

## Acceptance criteria

v0.7.7 is "shipped" when **all** of these pass:

- [ ] All 5 SQL adapters live + Vitest equivalent (pytest unit + integration tests)
- [ ] Each adapter produces structurally-valid `SecurityFinding` per row collected
- [ ] Each adapter's NIST mapping table is documented in `docs/sql-collectors.md`
- [ ] Each adapter ships its `BLIND_SPOTS` list (Python const + AR back-matter)
- [ ] Read-only principal verification works on each adapter (write-priv detection finding fires correctly)
- [ ] Air-gap mode validation works on each adapter
- [ ] Okta collector lives + Vitest equivalent
- [ ] ServiceNow integration lives + Vitest equivalent
- [ ] `docs/benchmarks.md` v2 published with new SQL-collector measurements
- [ ] All v0.7.x baselines hold: pytest 1100+ passing (965 + ~150 new SQL tests), mypy strict clean, ruff lint clean
- [ ] GitHub code-scanning open-alert count stays at v0.7.5 baseline (≤7 advisory findings)
- [ ] `pre-release-review` skill (Pre-tag variant) all 6 steps green
- [ ] `pypi-attestations verify pypi` + `gh attestation verify` green
- [ ] Local Docker build green pre-tag (now includes SQL adapter wheels)
