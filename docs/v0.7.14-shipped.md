## Headline shipments

### 🎉 Codecov P2.1 deep-dive RESOLVED

The v0.7.13 P0.3 `source_pkgs` fix produced correct full-path
Cobertura XML locally, but Codecov continued to show
`state: error` + `coverage: 0.0` + `report.files: []` on
c0c9a31, e32b742, 9613e62.

Investigation via the Codecov upload-state endpoint surfaced
the upload landing on their side but parsing to 0 files. Most
likely cause: the
`flag_management.individual_flags[].paths` glob
`["packages/*/src/"]` was filtering all files out (Codecov's
glob semantics for flag-paths apparently don't reliably recurse
into sub-directories of the matched prefix; the new full-path
filenames emitted by `source_pkgs` are deeper than 3 levels
under the glob root).

**Fix**: removed the `flag_management` block entirely.
Coverage flagging by `python` still works at the upload-time
`flags: python` arg in tests.yml. Carryforward defaults to
True per Codecov's docs without explicit config.

**Validation on v0.7.14 ship commit `7a120f3`**:
- Codecov state: `complete` (was `error`)
- Coverage: **82.14%** (was 0.0%)
- Files: **172** (was 0)
- Above the 80% Silver-tier `test_statement_coverage80` threshold

The Codecov badge on the README started rendering correctly
immediately post-push. No badge URL change needed; the
existing URL has been correct all along — only the dashboard
state was wrong.

### container-build.yml Wait extension (P2.2)

The v0.7.13 e32b742 (PR #18 merge) container-build smoke test
failed because the existing Wait step polled only the
`evidentia` umbrella package. Pip's resolution then failed on
`evidentia-core<0.8.0,>=0.7.13` because evidentia-core hadn't
propagated yet. The pin-trap fix correctly REFUSED to fall
back, surfacing the propagation race as a hard CI failure.

**Fix**: extended the Wait-for-PyPI step to poll all 6
inter-package deps before the docker build runs. Same pattern
as Step 7 G6 fresh-venv polling proved out. Closes the same
race that v0.7.6 P0.5 CI1 closed for the publish-container
path.

## What didn't work first try

### `flag_management` glob recursion

The Codecov fix attempt 1 was correct, but discovery required
querying the Codecov upload-state endpoint directly (the
GitHub Action's stdout doesn't expose processor errors).
Lesson: when upload telemetry shows success but state stays
error, query the Codecov API for raw processor diagnostics
before attempting more config tweaks.

### tailwind 4 migration deferred to v0.7.15

The 2-day time-box (per §23.B) was respected. v0.7.14 ships
without tailwind 4 + the migration lands as the dedicated
v0.7.15 P0.1 slice. The deferral was the correct call given:
- shadcn/ui preset rewrite is multi-day work
- `tailwindcss-animate` v1.0.7 is not v4-compatible (needs
  replacement with `tw-animate-css`)
- Visual diff validation requires manual inspection of 3 main
  routes

### Historical leak surfaced via standing-rule sweep

The v0.7.14 sweep on `v0.7.13..HEAD` flagged the literal
`Pasha walk-through (v0.9.0 design phase)` in the message
body of commit `9613e62` (made during v0.7.13 cycle post-ship
cleanup; already on origin/main).

**Disposition**: ACCEPT historical state. Force-push history
rewrite is destructive. Mitigation: per-commit standing-rule
sweep discipline tightened in v0.7.15 P0.3 (new pre-commit
hook scans staged files; commit-message body is a
documented limitation deferred to v0.7.16 if pattern recurs).
