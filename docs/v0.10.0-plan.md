# Evidentia v0.10.0 — implementation plan

> **Status**: PLANNED (drafted 2026-05-21). First release of the
> v0.10.x research-driven integration line (see
> [ROADMAP.md](ROADMAP.md) → "v0.10.x" and
> [integration-survey.md](integration-survey.md)).
>
> **Scope locked** (Allen, 2026-05-21, via planning Q&A):
> 1. **OCSF adoption** — *native model + OCSF mapping*: Evidentia keeps
>    its own Pydantic finding model (evolved from `SecurityFinding`,
>    retaining OLIR-typed control mappings + provenance) with a strict
>    bidirectional OCSF Compliance Finding mapping layer.
> 2. **0.10.0 contents** — the normalized schema, SARIF emit for
>    `evidentia gap`, and **3 pilot collectors** (AWS, GitHub, the
>    Postgres SQL adapter). The remaining ~11 collectors and the
>    third-party OCSF *ingestion* collector are deferred to v0.10.1.

---

## 1. Context

The 2026-05-21 competitive/integration research pass identified a
normalized, OCSF-aligned findings schema as the keystone of the
v0.10.x line: it is the change that makes downstream integrations
(Prowler, AWS Security Hub, Datadog, RegScale ingestion) cheap rather
than bespoke. v0.10.0 lays that foundation and ships the first
operator-visible payoff (SARIF emit).

**Correction to an earlier assumption.** The research framing said
"collectors emit framework-specific shapes." The code does not bear
that out — every collector already emits one unified model,
`SecurityFinding`. The real value of OCSF alignment is therefore
*interoperability*, not *internal normalization*: a finding model that
maps losslessly to/from the OCSF Compliance Finding class means any
OCSF-emitting tool can be ingested through a single adapter (v0.10.1),
and Evidentia findings can be exported as OCSF for SIEMs and other
consumers.

## 2. Goals & non-goals

**Goals (v0.10.0):**
- Evolve `SecurityFinding` into an OCSF-aligned normalized finding
  model — **additively and non-breakingly**.
- Ship a strict **bidirectional OCSF mapping layer** (`to_ocsf` +
  `from_ocsf`) targeting the OCSF Compliance Finding class.
- Refactor **3 pilot collectors** (AWS, GitHub, Postgres) to populate
  the evolved model fully and exercise the OCSF round-trip.
- Add a **SARIF 2.1.0 output format** to `evidentia gap`.

**Non-goals (deferred to v0.10.1+):**
- Refactoring the remaining ~11 collectors (Okta, the other 4 SQL
  adapters, Databricks, Snowflake, Vanta, Drata, BitSight,
  SecurityScorecard).
- The third-party **OCSF ingestion collector** (reads Prowler / AWS
  Security Hub OCSF output). The mapping layer's `from_ocsf` direction
  *is* built and tested in v0.10.0; only the collector that wires it to
  a file/API reader is deferred.
- MCP plugin / GRC Engineering Club marketplace, persona modes,
  YAML-driven catalogs, AI-governance regulatory packs — later v0.10.x.

is not frozen and the evolution is additive (new optional fields with

unchanged; `ControlGap`/`GapAnalysisReport` are untouched; a new CLI

## 4. Phase 0 — pre-implementation verification

Resolve before writing code; each is a quick lookup, not a build task.

- **OCSF schema version** — pin the exact current version at
  [schema.ocsf.io](https://schema.ocsf.io) (the March 2026 AWS source
  cited v1.8.0; confirm). The mapping layer binds to one version.
- **Compliance Finding vs Detection Finding** — confirm which OCSF
  class AWS Security Hub and Prowler actually emit for compliance
  posture. The generic `security_finding` was deprecated in OCSF
  v1.1; the **Compliance Finding** class (class_uid 2003) is the
  target, but Prowler reportedly emits Detection Finding — verify, as
  it shapes the `from_ocsf` ingestion design in v0.10.1.
- **`py-ocsf-models`** — evaluate coverage + maturity of
  [`prowler-cloud/py-ocsf-models`](https://github.com/prowler-cloud/py-ocsf-models)
  (Apache-2.0, Pydantic). Decision gate: if it models the Compliance
  Finding class adequately, depend on it for the OCSF side of the
  mapping layer; if it is Detection-Finding-only, hand-model a minimal
  OCSF Compliance Finding Pydantic subset instead. Either way the
  dependency is isolated to the mapping module — the core
  `SecurityFinding` model never imports it.
- **Findings persistence** — confirm collected `SecurityFinding`
  objects are not persisted as long-lived JSON anywhere (they flow to
  OSCAL AR via `oscal/exporter.py`). If they are, the additive
  evolution still round-trips, but note it.

**Phase 0 outcomes (resolved 2026-05-21):**
- *OCSF version* → the OCSF schema is at **1.8.0** (released
  2026-03-18). Note (resolved in Phase 2): the chosen library
  `py-ocsf-models` 0.9.x models the OCSF **1.5.0** schema — the
  Compliance Finding class is stable across OCSF 1.1+, so this is a
  version-label detail, not a functional gap.
- *Compliance vs Detection Finding* → Evidentia emits the **Compliance
  Finding** class (`class_uid` 2003) — semantically correct for
  control-pass/fail findings. **Prowler emits Detection Finding**, so
  the deferred v0.10.1 ingestion collector needs a Detection Finding
  path too (see §14).
- *`py-ocsf-models`* → **adopt it.** Apache-2.0, Pydantic, actively
  maintained (commit within 2 days of this check); it models the
  Compliance Finding class + the `compliance` object. Used for the
  OCSF side of the mapping layer (Phase 2), isolated to that module.
- *Findings persistence* → no managed finding store exists;
  `evidentia collect --output` writes ad-hoc JSON files. Additive
  model evolution re-reads them cleanly (Pydantic fills defaults). No
  migration module needed.

## 5. Phase 1 — the normalized findings schema

Evolve `SecurityFinding` **in place**, keeping the class name for
v0.10.0 (a rename to e.g. `Finding` is a v0.10.1 candidate once all
collectors are migrated, done via the api-stability deprecation policy
with a one-cycle re-export alias). All changes additive.

**New fields (all optional / defaulted → non-breaking):**
- `compliance_status: ComplianceStatus` — new enum
  (`PASS` / `FAIL` / `WARNING` / `NOT_APPLICABLE` / `UNKNOWN`),
  mirroring OCSF `compliance.status`. **Default `UNKNOWN`** — the
  Phase 0 collector audit found that while most collectors gather
  non-compliant (FAIL) items, the vendor-risk collectors (Vanta /
  Drata / BitSight / SecurityScorecard) also emit INFORMATIONAL
  inventory findings that are *not* failures. `UNKNOWN` is therefore
  the honest default for un-migrated construction sites; the 3 pilot
  collectors set it explicitly. This is the one genuine semantic gap
  vs. OCSF and is required for faithful `from_ocsf` ingestion
  (third-party OCSF carries passes).
- `remediation: str | None` — structured remediation text, maps to
  OCSF `remediation.desc`. `SecurityFinding` currently has none
  (`ControlGap` does, but not findings).
- Keep `control_mappings` (OLIR-typed) and `collection_context` exactly
  as-is — these are Evidentia's enrichment over OCSF and are retained
  per the locked decision.

**Explicitly not stored on the model**: OCSF envelope fields
(`class_uid`, `category_uid`, `activity_id`, `type_uid`) — these are
computed by the mapping layer, not persisted.

**Deliverable**: updated `models/finding.py`; new `ComplianceStatus`
enum; `models/__init__.py` export; docstring migration note following
the existing v0.7.0 precedent in the file.

## 6. Phase 2 — the OCSF mapping layer

New module: `evidentia_core/ocsf/` (package) — `finding_mapping.py`
with two pure functions plus the OCSF representation.

- `finding_to_ocsf(finding: SecurityFinding) -> dict` — emits an OCSF
  Compliance Finding object. Field mapping: `id` → `finding_info.uid`;
  `title`/`description` → `finding_info.title`/`desc`; `severity` →
  `severity_id`; `compliance_status` → `compliance.status` +
  `status_id`; `control_mappings` → `compliance.standards[]` +
  `compliance.requirements[]` (one per mapping; framework → standard,
  control_id → requirement); `resource_*` → `resource`;
  `collection_context` → `metadata` + `observables`; `remediation` →
  `remediation`; `first_observed`/`last_observed` → `time` fields;
  `raw_data` → `unmapped` or `raw_data`.
- `finding_from_ocsf(ocsf: dict) -> SecurityFinding` — the inverse.
  Built and round-trip-tested in v0.10.0 even though the ingestion
  *collector* that calls it is v0.10.1.
- OCSF representation — **resolved (Phase 0): depend on
  `py-ocsf-models`** (Apache-2.0, Pydantic, Prowler-team-maintained;
  models the Compliance Finding class). The dependency is isolated to
  this package — the core `SecurityFinding` model never imports it.
- Round-trip guarantee: `from_ocsf(to_ocsf(f))` preserves all
  Evidentia-native fields; OLIR relationship + justification ride in
  `compliance.standards[].requirements` or an OCSF extension so they
  survive export/import.
- Optional: `COLLECT_OCSF_EMITTED` EventAction (append-only enum
  addition, minor-safe) if OCSF export is to be audit-logged.

## 7. Phase 3 — refactor 3 pilot collectors

Pilot set: **AWS** (`aws/collector.py` + `access_analyzer.py`),
**GitHub** (`github/collector.py` + `dependabot.py`), **Postgres**
(`sql/postgres/collector.py`) — chosen as one representative of each
collector class (cloud API, code host, database).

Per collector: at every `SecurityFinding(**kwargs)` construction site,
populate the new fields — set `compliance_status` explicitly
(`FAIL` for the non-compliant items collectors gather today; the
model evolution also makes a future `PASS`-collecting mode possible)
and populate `remediation` where the source system provides it (AWS
Security Hub `Remediation.Recommendation`, GitHub Dependabot advisory
text, etc.). No change to the per-collector `mapping.py` tables. The
11 un-refactored collectors are untouched and keep working (additive
evolution). Add a pilot-collector OCSF round-trip assertion to each
collector's tests.

## 8. Phase 4 — SARIF emit for `evidentia gap`

Independent of Phases 1-3 — can proceed in parallel.

- New module `evidentia_core/gap_analyzer/sarif.py`:
  `gap_report_to_sarif(report: GapAnalysisReport) -> dict` producing a
  SARIF 2.1.0 log.
- Mapping: `tool.driver` = "Evidentia" + version + a `rules[]`
  catalog where each distinct control is a `reportingDescriptor`
  (`id` = `framework/control_id`, `name`, `fullDescription` =
  `control_description`, `help` = `remediation_guidance`); each
  `ControlGap` → one `result` (`ruleId`, `level` from `gap_severity`:
  critical/high → `error`, medium → `warning`, low/informational →
  `note`; `message` = `gap_description`).
- **GitHub code-scanning nuance**: code scanning expects `locations`
  with physical artifact URIs; a compliance gap is not file-bound.
  Emit a logical location (the control inventory file from
  `GapAnalysisReport.inventory_source`) plus `partialFingerprints`
  (hash of `framework`+`control_id`) so results are stable across
  runs. Document that the SARIF is also valid as a standalone artifact
  independent of code scanning.
- Wire a `sarif` choice into `evidentia gap`'s output-format selection
  (`cli/gap.py` + `gap_analyzer/reporter.py`) alongside
  `json/csv/markdown/oscal`. Adding the format is non-breaking
  (api-stability §3).

- **Collector tests**: extend AWS / GitHub / Postgres collector tests
  for the new fields + an OCSF round-trip assertion.
- **Integration**: `evidentia gap --output sarif` end-to-end against a

- **Gates**: mypy strict 0/0 across all touched packages; ruff clean;

- **Docs**: new `docs/ocsf-mapping.md` (the field-by-field mapping

## 10. Sequencing & dependencies

```
Phase 0 (verification) ─┬─> Phase 1 (schema) ──┬─> Phase 2 (OCSF mapping)
                        │                      └─> Phase 3 (pilot collectors)
                        └─> Phase 4 (SARIF) ───────> [independent]
Phase 5 (tests/docs) runs continuously alongside 1-4.
```

Phase 4 (SARIF) has no dependency on the schema work and can be built
first or in parallel — a good early, shippable win. Phases 2 and 3
both depend on Phase 1's model. Rough size: a focused multi-day effort
per phase; Phase 1 + 2 are the critical path.

## 11. Risks & mitigations

| Risk | Mitigation |
|---|---|
| OCSF Compliance Finding under-modeled by `py-ocsf-models` | Phase 0 gate; fall back to a hand-modeled Pydantic subset isolated to `evidentia_core/ocsf/`. |
| OLIR relationship + justification have no clean OCSF home | Carry them in `compliance.standards[].requirements` detail or a namespaced OCSF extension; round-trip test enforces no loss. |
| `compliance_status` default misclassifies a finding | **Resolved (Phase 0)**: default is `UNKNOWN`, not `FAIL` — the vendor-risk collectors emit informational inventory findings, so `FAIL` would be wrong. Pilot collectors set the value explicitly; the 11 legacy collectors get `UNKNOWN` until migrated in v0.10.1. |
| OCSF schema version churn | Native model is insulated by design; the mapping layer pins one OCSF version and is the only thing that moves on an OCSF bump. |
| SARIF results without physical locations degrade the GitHub code-scanning UX | Logical location + `partialFingerprints`; document standalone-artifact use as the primary contract. |
| Scope creep into the other 11 collectors | Hard line: 3 pilots only; the rest are a tracked v0.10.1 item. |

## 12. End-to-end verification

1. `evidentia collect aws|github` (or fixtures) → confirm pilot
   collectors emit findings with `compliance_status` + `remediation`
   populated.
2. Round-trip: `finding_from_ocsf(finding_to_ocsf(f)) == f` for pilot
   findings (property test).
3. Validate `to_ocsf` output against the OCSF schema for the pinned
   version.
4. `evidentia gap --inventory <fixture> --output sarif` → validate the
   file against the SARIF 2.1.0 schema; spot-check it renders in
   GitHub code scanning on a test push.
5. Full suite green; mypy strict 0/0; ruff clean; OSCAL trestle
   round-trip green.
6. Pre-release-review per `docs/release-checklist.md` before tagging.

## 13. Release mechanics

- Bump all 8 `pyproject.toml` files atomically via
  `scripts/bump_version.py --to 0.10.0` (updates the inter-package
  pin ranges; see `docs/release-checklist.md`).
- 0.9.9 → 0.10.0 is the correct minor bump (meaningful new feature
  surface, per the project versioning rule).
- Tag-driven release: `git tag v0.10.0 && git push origin v0.10.0`
  fires `release.yml`. **Tagging + push require Allen's explicit
  approval** per the publishing-authority protocol — this plan does
  not pre-authorize it.
