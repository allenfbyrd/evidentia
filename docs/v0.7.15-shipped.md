## Headline shipments

### Tailwind 3 → 4 migration

The deferred half of PR #21. Full shadcn/ui new-york preset
rewritten from JS-config (`tailwind.config.ts`) to CSS-first
`@theme {}` blocks in `src/index.css`. PostCSS chain replaced
with the first-class `@tailwindcss/vite` Vite plugin.
`tailwindcss-animate@1.0.7` (last v3-era release; not
v4-compatible) replaced with `tw-animate-css@1.4.0` (community
fork; v4-native).

The migration was surprisingly mechanical: shadcn/ui's preset
is mostly CSS-variable-driven already, so most theme tokens
lifted straight into `@theme {}` as `--color-*` declarations
referencing the existing `:root` + `.dark` HSL CSS variables.
The dark-mode strategy required one v4-specific addition:
`@custom-variant dark (&:where(.dark, .dark *))` — without
this directive, Tailwind 4 defaults to media-query dark-mode
which doesn't honor the shadcn/ui `<html class="dark">` toggle
pattern.

Visual output verified unchanged: severity palette, dark-mode
toggle, accordion animations all carry forward. CSS bundle
size grew from 22.20 KB (v3) → 35.04 KB (v4); gzipped from
4.98 KB → 6.41 KB. The growth is Tailwind 4's broader default
emit (more utilities included by default), not new application
code.

### SettingsPage.tsx refactor

The v0.4.1 `useEffect` + `setState` pattern that copied
`configQuery.data` into local form state replaced with a
**key-based remount of a new `<SettingsForm/>` sub-component**:

- Parent `<SettingsPage/>` owns the queries. Renders the inner
  form ONLY when `configQuery.data` has resolved.
- Inner `<SettingsForm/>` is mounted with
  `key={config.source_path ?? "default"}`. Each new
  config-load triggers a remount with fresh state seeded by
  `useState` lazy initializers from props.

This is React's canonical idiom for "initialize state from
async data". `react-hooks/set-state-in-effect` lint rule
promoted from `warn` (v0.7.14 documented-deferral) → `error`
(v0.7.15 enforced); any future regression now fails CI.

### Standing-rule sweep pre-commit hook

`scripts/standing_rule_sweep.sh` + `.pre-commit-config.yaml`
local hook close the gap that produced the v0.7.13-cycle
commit-message-body leak. The sweep ran only at pre-push
before; v0.7.15 extends to pre-commit (file-content scan on
staged content).

Limitation documented inline + in `docs/v0.7.14-shipped.md`:
commit-message body is NOT scanned by the file-content hook.
v0.7.16 adds the `commit-msg` stage variant that closes that
gap too.

## What didn't work first try

### release.yml's publish-container Wait step had the same single-package gap

The v0.7.15 first publish-container fire FAILED at the docker
build with the same shape as the v0.7.13 e32b742 propagation
race: `evidentia==0.7.15` was visible on PyPI when the Wait
step ran, but pip's resolution then failed on
`evidentia-core<0.8.0,>=0.7.15` (not yet propagated). The
pin-trap fix correctly REFUSED to fall back, surfacing the
race as a hard CI failure.

**Recovery on v0.7.15 ship**: `gh run rerun --failed` 5 min
later when all 6 had propagated. The structural fix landed
post-ship as commit `fd36e78` (extends release.yml's
publish-container Wait step to poll all 6 inter-package deps,
matching the v0.7.14 P2.2 fix that already extended
container-build.yml's Wait step).

This was the LAST PyPI propagation race surface in the v0.7.x
release pipeline. Both the smoke-test path
(container-build.yml; v0.7.14 P2.2) and the publish path
(release.yml; v0.7.15 post-ship hardening) now poll all 6
packages.

### Tailwind 4 migration finished in <1 hour, not 2 days

The §23.B 2-day time-box was conservative. Actual work fit in
under 1 hour because (a) shadcn/ui's preset is mostly
CSS-variable-driven, (b) `tw-animate-css` is a drop-in
replacement for `tailwindcss-animate`, (c) the v4 vite plugin
is a clean replacement for the PostCSS chain.

Lesson: time-boxes for migrations should be set per-repo based
on actual customization depth, not generic estimates.
