## 6. Error handling

- **C1 catalog load test**: existing `tests/unit/test_catalogs/test_all_bundled.py` smoke suite auto-extends (parametrized) — new YAMLs must pass the smoke or C1 reverts.
- **C1 OSCAL validation**: pipeline runs `oscal-cli validate` (or Trestle) against `osps-baseline.oscal.json` — must pass or C1 reverts. (Tool availability check at execution time; if `oscal-cli` isn't on PATH, install via the Trestle pinned version and document in commit body.)
- **C3 CI gate failure modes**: any cited evidence link 404 → workflow fails; the conformance doc is intentionally fragile-to-link-rot so it cannot drift silently.
- **C5 crosswalk schema**: a unit test loads all 5 YAMLs against the inline `schema_version: 1` shape; any mapping row missing `source_control` or `target_controls` fails.
- **C6 collector retry/backoff**: reuses existing `evidentia-collectors.github` retry+backoff + cursor-pagination patterns from v0.10.5 idempotency hardening.
- **C7 workflow permissions audit**: writes a report to stdout; doesn't fail the build (advisory only this cycle; v0.10.7 can add a CI gate per §2.E).
- **GHSA enablement (T1, inside C2)**: idempotent — re-applying the PATCH is safe if it's already enabled. Pre-flight `gh api repos/.../security-advisories` returns-empty check captured in commit body.
- **OSCAL upstream PR (T2)**: separate publishing action; v0.10.6 ships independent of PR-merge timing.

**What the brainstorm-locked decision said** (after the 2026-05-26 review): "YAML in `packages/evidentia-core/src/evidentia_core/crosswalks/data/` with inline `schema_version: 1` + `provenance` + `verification` fields. Matches the existing catalog convention."

**What the codebase actually has**:

- Existing 8 crosswalks at `packages/evidentia-core/src/evidentia_core/catalogs/data/mappings/` (note: under `catalogs/`, not under a new top-level `crosswalks/`)
- Format: **JSON**, not YAML
- Loader: `packages/evidentia-core/src/evidentia_core/catalogs/crosswalk.py` — uses `import json` directly
- Schema: existing `CrosswalkDefinition` Pydantic model with `source_framework`, `target_framework`, `version`, `generated_at`, `source` (string-typed provenance), and `mappings[]` of `(source_control_id, source_control_title, target_control_id, target_control_title, relationship, notes)`

**Load-bearing assumption that failed**: "catalogs use YAML so crosswalks should too." True that the 89 bundled catalogs are YAML files; the crosswalk SUBSURFACE has been JSON since at least v0.7.x. The earlier review's "matches catalog convention" rationale conflated the catalog data files with the crosswalk data files.

**How it was caught**: at the start of the writing-plans skill, the file-structure exploration opened `packages/evidentia-core/src/evidentia_core/catalogs/` and surfaced `crosswalk.py` (the loader module) + `data/mappings/` (the existing data directory). Reading `head -30` of `crosswalk.py` revealed `import json` and the existing schema. Listing `data/mappings/` revealed 8 in-tree `*.json` files.

**Corrected plan** (Option 2 chosen by Allen 2026-05-26): keep JSON; reuse `catalogs/data/mappings/`; **additively** extend `CrosswalkDefinition` in `evidentia_core.models.catalog` with 3 optional fields (`provenance`, `verification`, `verification_note`) for the OSPS-attested disclaimer shape. Backward-compatible per api-stability frozen-surface contract (existing 8 crosswalks unchanged — new fields default to `None`).

**Generalized lesson for future brainstorms**: when locking a design decision that asserts "matches existing pattern," OPEN the existing files; don't just `ls` the parent directory. The 2026-05-26 brainstorm-review's "matches catalog convention" claim was based on incomplete data (`ls packages/evidentia-core/src/evidentia_core/catalogs/data/international/` showed `.yaml` files; that's correct for catalogs but doesn't generalize to crosswalks). The standing-directive memory's triple-validation rule names this exact failure mode: "ALWAYS verify with a direct source ... before asserting in a doc, plan, or chat. Memory entries about external state decay fast." Same applies to memory entries about INTERNAL state.

### 12.2 — OSPS Baseline control counts 21/38/58 → 25/42/63 (caught 2026-05-26 during Phase 1 implementation)

**What the brainstorm-locked plan said** (Task 1.2 / 1.3 / 1.4): "Maturity 1: 21 controls, Maturity 2: 38 controls, Maturity 3: 58 controls."

**What the pinned upstream tarball actually has** (`ossf/security-baseline@ac6bbec8aecf51dce41f62712745f7949ab6bdeb`, May 12, 2026):

- Top-level controls: M1 = 17, M2 = 32, M3 = 40
- Assessment-requirements (per-requirement granularity, the meaningful per-maturity atom): M1 = 25, M2 = 42, M3 = 63
- Total: 41 top-level controls (the plan's "41" was correct at the overview level)

The plan's 21/38/58 numbers don't match upstream at any granularity. Source of the drift: the v0.10.5-plan.md §3 Phase 1 carried those numbers from the original integration-plan audit (2026-05-24), which appears to have estimated them without verifying against the actual upstream YAML files.

**Load-bearing assumption that failed**: "the integration-plan audit numbers are upstream-verified."

**How it was caught**: at Task 1.2 Step 3 implementation (authoring `osps-baseline-m1.yaml`), the implementer fetched the upstream tarball at the pinned SHA per Task 1.1 and counted assessment-requirements by maturity-field. The count did not match the plan. Standing-directive triple-validation discipline obligated the implementer to surface the drift rather than paper over.

**Corrected plan**: Phase 1's test assertions + commit body use the upstream-verified counts (25 / 42 / 63 at assessment-requirement granularity, which is the granularity the per-maturity YAMLs adopt because per-maturity applicability is per-requirement upstream). The OSCAL sidecar uses top-level-control granularity (41 controls). The granularity choice is well-documented in `ea9f117`'s commit body.

**Two-of-two pattern**: this is the second instance of the §12 pattern (count drift after the §12.1 crosswalk format drift). Both came from brainstorm-locked decisions that asserted facts about external/internal state without verifying. **Cycle-level lesson**: every brainstorm-locked numeric claim or schema/format claim needs upstream verification before it lands in a plan doc, not just at execution time. Future v0.10.7+ brainstorms should add a "data-claim verification" step before the spec self-review.

### 12.3 — Crosswalk source data location (caught 2026-05-27 during Phase 5 implementation)

**What the brainstorm-locked Phase 5 plan said** (impl-plan §Phase 5 Task 5.2): "walk the upstream OSPS Baseline catalog data at `packages/evidentia-core/src/evidentia_core/catalogs/data/international/osps-baseline-m{1,2,3}.yaml` ... For each OSPS control, check its `guidelines[]` array for entries that name the target framework."

**What the codebase actually has**: the 3 C1-bundled per-maturity YAMLs at `catalogs/data/international/osps-baseline-m{1,2,3}.yaml` do NOT carry a `guidelines[]` array per control. C1's per-maturity flattening converted the upstream per-control structure into per-assessment-requirement entries and intentionally dropped the `guidelines[]` block (each assessment-requirement carries a single `guidance` string, not the rich SSDF/CSF/CRA/PCIDSS/800-161 reference-id structure upstream's per-family YAMLs ship). The C1 commit `ea9f117` body documents this granularity choice (assessment-requirement vs top-level control) but does not call out the guidelines[] drop, which is what tripped the Phase 5 implementer.

**Load-bearing assumption that failed**: "the C1-bundled per-maturity YAMLs preserve upstream's per-control guidelines[] reference-id structure."

**How it was caught**: at Task 5.2 Step 1 implementation, the C5 implementer opened `osps-baseline-m1.yaml` looking for `guidelines:` blocks to extract NIST SSDF / NIST CSF / EU CRA / PCI DSS / NIST 800-161 reference-ids. The blocks were not present in the bundled YAML. The implementer triple-validated by also checking m2 + m3 files (same result), then verified the upstream YAMLs at the pinned SHA `ac6bbec` do carry the `guidelines[]` blocks per-family (OSPS-AC.yaml, OSPS-BR.yaml, etc.). Standing-directive triple-validation obligated reporting the situation rather than fabricating mappings.

**Corrected plan**: source the 5 OSPS crosswalk JSONs from the upstream OSPS per-family YAMLs at the same pinned SHA `ac6bbec8aecf51dce41f62712745f7949ab6bdeb` that C1 used. Zero SHA drift between C1 and C5 — both anchor to the same upstream commit. The bundled per-maturity YAMLs remain the canonical Evidentia-runtime source for per-maturity control evaluation (their original intent); the upstream per-family YAMLs are the canonical source for the guidelines[] reference-ids that drive crosswalk authoring. This split is clean: maturity-granularity for runtime + control-granularity for crosswalk authoring, both at the same SHA. Documented in commit `e443fa5`'s body.

**Three-of-three pattern**: every §12 entry so far has caught a different load-bearing assumption that didn't survive verification: §12.1 a SCHEMA-LOCATION claim ("YAML in new dir"), §12.2 a NUMERIC-COUNT claim ("21/38/58 top-level controls"), §12.3 a STRUCTURAL-PRESERVATION claim ("C1 bundle preserves upstream's guidelines[]"). **Generalized cycle-level lesson, updated**: brainstorm-locked claims about (a) where data lives, (b) how much data exists, AND (c) what structure existing data has all need upstream verification before they land in a plan. The Phase B audit-synthesis cycle (per the [[release_pattern_audit_directive]] memory) should consider whether the writing-plans skill needs a "data-shape verification" pre-write check sitting alongside the existing "spec-self-review" step.

**Maintenance follow-up logged for v0.10.7**: the C5 code-quality reviewer flagged that future upstream OSPS bumps require a manual regeneration pass across 5 files + ~15 hardcoded SHA references. A `scripts/catalogs/gen_osps_crosswalks.py` script (or a `docs/crosswalk-regeneration.md` recipe) would close the gap. Not a v0.10.6 blocker; tracked in v0.10.7 PLANNED.

### 12.4 — OSPS control-ID granularity (caught 2026-05-27 during Phase 6 implementation)

**What the brainstorm-locked Phase 6 plan said** (impl-plan §Phase 6 Task 6.1): "Plan's suggested coverage (validate against actual OSPS control titles): AC family: OSPS-AC-01 (branch protection), OSPS-AC-02 (signed commits), OSPS-AC-03 (required reviews)..." The plan enumerated 15 controls using **top-level family IDs** (`OSPS-AC-01`, `OSPS-VM-05`, etc.).

**Load-bearing assumption that failed**: "OSPS controls and OSPS assessment-requirements use the same ID granularity." (The same-granularity-everywhere assumption that §12.2 also disproved, here re-tripped at the helper-naming stage.)

**How it was caught**: at Task 6.1 enumeration, the C6 implementer walked the C1-bundled per-maturity YAMLs to confirm which OSPS items had GitHub-API-automatable signals. The walk surfaced that the testable items are at the `.ZZ` granularity (the upstream per-family YAMLs at the pinned SHA `ac6bbec` use the same atomization). The implementer used the pre-brief's explicit "use the real ids; do not fabricate" directive to pivot the 15-helper enumeration onto real `OSPS-XX-YY.ZZ` IDs.

Documented in commit `09bf498`'s body.

**Four-of-four pattern + tightened cycle-level lesson**: §12.1 = where data lives; §12.2 = how much data exists; §12.3 = what structure data has; §12.4 = at what granularity to address data. Every §12 entry has caught a different load-bearing assumption that didn't survive verification. **Future v0.10.7+ brainstorms should make all four checks explicit before locking spec language about external data: (a) location, (b) cardinality, (c) shape, (d) addressing-granularity.** This pattern itself is now load-bearing — the Phase B audit-synthesis cycle should consider promoting it from corrections-log into the writing-plans skill's pre-write checklist.
