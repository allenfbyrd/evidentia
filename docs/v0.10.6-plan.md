## 4. Architecture / components

### 4.1 — OSPS Baseline catalog (C1)

Three new files in `packages/evidentia-core/src/evidentia_core/catalogs/data/international/`:

- `osps-baseline-m1.yaml` — 21 controls (any project)
- `osps-baseline-m2.yaml` — 38 controls (≥2 maintainers + small consistent user base; solo-dev honest-gap footnote)
- `osps-baseline-m3.yaml` — 58 controls (large consistent user base; partial coverage; honest-gap declarations for OSPS-AC-04.01/02, OSPS-GV-04, OSPS-QA-07)

Pattern: matches existing `fedramp-rev5-{low,moderate,high}.yaml` + `cmmc-2-{l1,l2,l3}.yaml` 3-catalog flatten. Zero schema change. Tier-A bundling (Apache-2.0 verbatim).

OSCAL conversion: `osps-baseline.oscal.json` in same dir; NIST OSCAL Catalog v1.2.1 (Evidentia's current). Clean-room transform from upstream YAMLs to OSCAL Catalog shape; preserves every OSPS control + assessment-requirement + maturity-level applicability via OSCAL `props` extension pattern.

### 4.2 — OSPS-CONFORMANCE.md + CI gate (C4)

Three artifacts:

- `OSPS-CONFORMANCE.md` at repo root — Evidentia's self-attested conformance claim against OSPS Baseline v2026.02.19 Level 2 with partial Level 3. Per-control evidence table; honest-gap section for the 3 structurally-unreachable-solo controls.
- `.local/pre-release-review/osps-conformance.yaml` — machine-readable companion (per-control `PASS | FAIL | HONEST_GAP` + evidence pointers). Gitignored; lives skill-side per RF3.
- `.github/workflows/verify-osps-conformance.yml` — new CI workflow that re-validates each claimed-PASS control's evidence link via `gh api` calls; fails if any cited evidence 404s.

### 4.5 — 5 OSPS crosswalks (C6)

New directory `packages/evidentia-core/src/evidentia_core/crosswalks/data/` (if not extant — check during execution; might reuse `docs/crosswalks/`). Five JSON files:

- `osps-baseline_to_nist-ssdf-800-218.json`
- `osps-baseline_to_nist-csf-2.0.json`
- `osps-baseline_to_eu-cra.json`
- `osps-baseline_to_pci-dss-4.0.json`
- `osps-baseline_to_nist-800-161.json`

Each file structure:

```json
{
  "$schema": "https://evidentia-grc.dev/schemas/crosswalk/v1.json",
  "provenance": "upstream-osps-guidelines",
  "verification": "self-attested-via-upstream",
  "verification_note": "Mappings auto-extracted from the OpenSSF OSPS Baseline `guidelines[]` array at <upstream-sha>. Not independently verified against the target framework. Consumers requiring independent verification should plan a hand-check pass.",
  "source_framework": "osps-baseline-v2026.02.19",
  "target_framework": "...",
  "mappings": [
    {"source_control": "OSPS-AC-01", "target_controls": ["..."], "upstream_reference": "..."}
  ]
}
```

Single mapping schema; consumers can hand-verify later by changing `verification: self-attested-via-upstream` → `verification: hand-checked-2026-MM-DD`.

### 4.6 — GitHub collector OSPS extension (C7)

Extends `packages/evidentia-collectors/src/evidentia_collectors/github/` to cover ~15 OSPS controls across:

- **AC family** (Access Control): branch-protection settings, required reviews, signed commits, workflow `permissions:` parse
- **BR family** (Build & Release): release-tag signing, attestation presence, SLSA provenance
- **GV family** (Governance): CODEOWNERS, contributing.md presence, governance.md presence
- **VM family** (Vulnerability Management): Dependabot enabled (reuses existing v0.9.x infrastructure for VM-05), secret scanning enabled, code scanning enabled

Each control gets a `populate_osps_<id>(...)` helper that maps GitHub API state → `ComplianceStatus` + evidence path.

### 4.7 — Workflow permissions audit (C8)

`scripts/audit_workflow_permissions.py` — greps `.github/workflows/*.yml` for missing/loose `permissions:` blocks. Reports per-workflow:

- ✅ explicit top-level `permissions: read` + per-job elevations
- ⚠️ missing top-level `permissions:` block (defaults to broad)
- ❌ explicit `permissions: write-all` or equivalent

Partial OSPS-AC-04.01/02 closure (the "explicit CI/CD permissions audit" maturity-3 control; full closure needs the "least-privilege" verification which is harder to automate).

## 6. Error handling

- **C1 catalog load test**: existing `tests/unit/test_catalogs/test_all_bundled.py` smoke suite auto-extends (parametrized) — new YAMLs must pass the smoke or C1 reverts.
- **C1 OSCAL validation**: pipeline runs `oscal-cli validate` (or Trestle) against `osps-baseline.oscal.json` — must pass or C1 reverts.
- **C4 CI gate failure modes**: any cited evidence link 404 → workflow fails; the conformance doc is intentionally fragile-to-link-rot so it cannot drift silently.
- **C6 crosswalk schema**: a unit test loads all 5 JSONs against the schema; any mapping row missing `source_control` or `target_controls` fails.
- **C7 collector retry/backoff**: reuses existing `evidentia-collectors.github` retry+backoff + cursor-pagination patterns from v0.10.5 idempotency hardening.
- **C8 workflow permissions audit**: writes a report to stdout; doesn't fail the build (advisory only this cycle; v0.10.7 can add a CI gate).
- **GHSA enablement (T1)**: idempotent — re-applying the PATCH is safe if it's already enabled.
- **OSCAL upstream PR (T2)**: separate publishing action; v0.10.6 ships independent of PR-merge timing.
