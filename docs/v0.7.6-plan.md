## Theme

**"UI alpha.2 completion + research-resync + benchmark design +
quickstart polish."**

v0.4.0 shipped the alpha.1 read-only web UI; the interactive
onboarding wizard, Gap Analyze form, Gap Diff picker, and
Risk Generate streaming page were planned for alpha.2 but
never landed (other priorities took precedence through v0.7.x).
v0.7.6 closes that loop — the GUI gets to feature-complete for
the v0.7 minor line — alongside the deferred quarterly research
re-sync (DOC5 from v0.7.3) if the Q3 cadence has arrived, plus
the performance benchmarking design + measurement work, the
90-second quickstart tutorial, and the deep `/security-review`
threat-model walk.

This is a polish + completion release. No new collectors, no new
output integrations, no new catalog tiers. Everything here is
finishing already-claimed surface. Realistic ship target: 4-5
weeks after v0.7.5.

## P0.7 — Code-scanning vulnerability triage (NEW 2026-05-01)

5 NEW HIGH alerts surfaced post-v0.7.5 push. All CodeQL / Scorecard false positives requiring annotation or dismissal, NOT runtime vulnerabilities:

| Alert | Rule | Path | Disposition |
|---|---|---|---|
| **#75** | TokenPermissionsID | `release.yml:211` (`contents: write` on `publish-container`) | **Accept** — `contents: write` is required for `softprops/action-gh-release` to append release notes. Same pattern as pre-existing alerts #29, #30 on the `publish-pypi` job. **Action**: dismiss as `won't-fix` with rationale; document in `docs/enterprise-grade.md` accepted-Scorecard-findings appendix. |
| **#74** | PinnedDependenciesID | `Dockerfile:62` (`evidentia[gui]==0.7.5`) | **Accept** — `==X.Y.Z` is already an exact PyPI pin; the alert wants `--hash=sha256:...` style which is overkill for a Trusted-Publisher-published wheel with PEP 740 attestation. **Action**: dismiss as `won't-fix` with rationale. |
| **#73** | py/path-injection | `security/paths.py:91` (the `validate_within` helper) | **False positive** — this IS the sanitizer; CodeQL doesn't recognize it. **Action**: contribute a CodeQL model that classifies `validate_within` as a sanitizer (long-term) OR add `# lgtm` suppression comment (short-term). |
| **#72, #71** | py/path-injection | `gap_store.py:182, 185` (downstream of `validate_within(candidate, store)` on line 181) | **False positive** — same root cause as #73; CodeQL flow analysis doesn't follow custom sanitizers. **Action**: dismiss with rationale referencing the upstream `validate_within` call site, or add the CodeQL custom-pack model. |

**Action plan**:
1. Add `.github/codeql/sanitizers.qll` declaring `validate_within` as a sanitizer for `py/path-injection`. Wire into `.github/codeql/codeql-config.yml`. Closes #71/#72/#73 properly.
2. Dismiss #74 + #75 as `won't fix` with rationale via `gh api repos/allenfbyrd/evidentia/code-scanning/alerts/<n> -X PATCH -f state=dismissed -f dismissed_reason=won_t_fix -f dismissed_comment=...`. **Each dismissal needs explicit Allen approval** (publish-authority gated).
3. Document accepted Scorecard findings (TokenPermissions × 3, PinnedDependencies × 1) in a new `docs/enterprise-grade-accepted-findings.md` appendix to keep `docs/enterprise-grade.md` clean.

**Estimated alert delta after triage**: 16 → 6 (12 HIGH → 6 HIGH after dismissing 5 + the CodeQL custom-pack closing 3 of the 12 HIGH; plus the existing 4 advisory Scorecard findings = ~6-7 remaining). Hits the v0.7.5 plan target of ≤7 retroactively.
