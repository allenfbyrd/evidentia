## Headline shipments

### Codecov source_pkgs fix (P0.3)

Root cause re-diagnosed against local `coverage.xml`: the v0.7.12
P0.7 `relative_files = true` fix didn't help because the Cobertura
XML format encodes file paths relative to whichever `<source>`
root matches, and 6 source roots × filename collisions (each
package has its own `__init__.py`, several have `app.py` etc.)
made the output fundamentally ambiguous.

**Fix**: switched `[tool.coverage.run]` from
`source = [<6 directory paths>]` to `source_pkgs = [<6
importable package names>]`. Cobertura output now records full
on-disk paths (`packages/evidentia-core/src/evidentia_core/...`)
without per-source-root disambiguation.

Local pytest produces 82.10% coverage measured against the
corrected scope. Codecov dashboard validation deferred to
v0.7.14 P2.1 (ultimately required additional config tuning —
see "What didn't work first try" below).

### release.yml CHANGELOG auto-population (P2.2.1) — STRUCTURAL FIX

Before v0.7.13: 9 GitHub Releases (v0.7.5 → v0.7.12) shipped
with 776-byte stub bodies — only the auto-appended container-
image stanza, no summary / headline-features / PEP 740 verify
stanza. Root cause: `release.yml` line 192 (Attach SBOM step)
created the release with no `body:` argument; only the
publish-container append-step added content.

**Fix**:
- `scripts/extract_changelog_block.py` (~100 LOC + 21 self-tests)
  parses CHANGELOG.md for the `[X.Y.Z]` block + wraps it in
  PEP 740 wheel-verify template + CHANGELOG-link footer.
- New `Extract release body from CHANGELOG` step in `release.yml`
  runs before the SBOM-attach step + writes `release-body.md`.
- `Attach SBOM to GitHub Release` step gains
  `body_path: release-body.md`.
- Sanity gate: refuses to publish if the body is < 1500 bytes.

**Validation end-to-end**: v0.7.13 release body landed at 6928
bytes on first publish (vs. 776-byte stub baseline). FIRST
release where the body is substantive on first publish.
Future releases never re-encounter the stub-body gap.

Plus 10 historical release-body backfills landed retroactively:
v0.7.5–v0.7.12 (9 stubs) replaced + v0.7.1 PEP 740 verify
stanza appended. All v0.7.x release bodies now substantive
(range: 2.4KB v0.7.3 → 28KB v0.7.9; canonical PEP 740 +
cosign verify stanzas across the set).

## What didn't work first try

### PyPI propagation race during release pipeline

The `release.yml` first run failed at the container-build step:
`evidentia==0.7.12` was on PyPI but `evidentia-ai==0.7.12`
hadn't propagated yet; the v0.7.12 P0.5 pin-trap fix correctly
refused to fall back to `evidentia-ai==0.7.11`. Fix: rerun
`gh run rerun --failed` 5 minutes later when PyPI fully
propagated. Validated the pin-trap fix working as designed.

The `container-build.yml` smoke test (separate workflow) also
failed on the post-push e32b742 commit for the same reason.
v0.7.14 P2.2 fixes this structurally by extending the Wait
step to poll all 6 packages.

### Codecov state error persisted post-source_pkgs fix

The v0.7.13 P0.3 `source_pkgs` fix was correct at the local XML
level (verified `<class filename="packages/evidentia-core/src/
evidentia_core/audit/logger.py">`), but Codecov continued to
show `state: error` + `coverage: 0.0` + `report.files: []` on
c0c9a31 + e32b742 + 9613e62.

Investigation via the Codecov upload-state endpoint surfaced
the upload landing on their side but parsing to 0 files. Most
likely cause: the `flag_management.individual_flags[].paths`
glob `["packages/*/src/"]` was filtering all files out
(Codecov's glob semantics for flag-paths apparently don't
reliably recurse into sub-directories of the matched prefix).

v0.7.14 P2.1 attempt 1 removes the `flag_management` block
entirely. If unsuccessful: attempt 2 (explicit `path_fixes:`),
then escalation per §23.A (switch to GitHub-built-in coverage).

### CodeQL alert recurrence pattern

Each Dockerfile-line bump creates a new alert ID for the same
Scorecard PinnedDependencies false-positive (#100 dismissed in
v0.7.12; #101 surfaced post-v0.7.13 push; #102 surfaced
post-PR-#18 merge). Per the runbook in
`docs/dockerfile-pinning.md`, dismiss-with-rationale is the
operational pattern; structural fix (hash-pinned
`requirements.txt`) is v0.8.0 G4-paired.

v0.7.14 P1.5 ships the hash-pinned `docker/requirements.txt`
generation tooling as v0.8.0 G4 foundation; the Dockerfile
install line continues to use exact-version pinning until
v0.8.0 G4 validates the full extras matrix.
