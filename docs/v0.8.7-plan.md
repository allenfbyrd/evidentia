### Phase 2 — `--faithfulness-threshold-mode` CLI flag

Closes the v0.8.6 P3 CLI-surface deferral. Library +
`resolve_threshold()` helper shipped in v0.8.6 P3; v0.8.7 P2
closes the operator-facing CLI surface.

Changes to `packages/evidentia/src/evidentia/cli/eval.py`:

- **Default change** `--faithfulness-threshold 0.3 → None`
  (sentinel for "user did not pass"). Backward compatible:
  callers who explicitly pass `--faithfulness-threshold 0.3`
  see identical behavior; callers who relied on the implicit
  default now get framework-aware resolution (an improvement,
  not a regression).
- **NEW** `--faithfulness-threshold-mode {framework-aware,
  fixed}` flag (default `framework-aware`). Validation:
  invalid value → exit 2.
- **Resolution precedence**:
  1. Explicit `--faithfulness-threshold` value always wins.
  2. `framework-aware` mode + `check_faithfulness=True` +
     samples non-empty → extract framework from first sample's
     `prompt_id` (canonical `<framework>:<control_id>` format)
     + `resolve_threshold(framework, method)`.
  3. `fixed` mode → `DEFAULT_FAITHFULNESS_THRESHOLD` (0.30)
     framework-agnostic.
- **Stdout summary** adds `faithfulness threshold: X.XX
  (<source>)` line where source is `explicit` /
  `framework-aware (framework=...)` / `fixed (framework-
  agnostic default)`.

3 new CLI tests in `TestFaithfulnessThresholdMode`:

- `test_invalid_mode_exits_2`: unknown mode → exit 2.
- `test_fixed_mode_uses_0_30`: fixed mode → harness gets 0.30
  (verified via stdout).
- `test_framework_aware_mode_uses_nist_0_60`: documents that
  bare framework identifier `"nist-800-53"` maps to 0.60 in
  `DEFAULT_THRESHOLDS_BY_FRAMEWORK_JACCARD` but Rev5 prefix
  `"nist-800-53-rev5"` doesn't match + falls back to 0.30
  framework-agnostic. Operators using the bare framework
  identifier get framework-aware defaults; operators using
  Rev5 prefix can pass `--faithfulness-threshold` explicitly
  OR use `--faithfulness-threshold-mode fixed`.
