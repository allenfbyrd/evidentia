## 2. Theme — "OSS first-mover artifacts"

v0.10.5 is positioned as the release where **Evidentia ships 4
first-of-its-kind OSS artifacts**, each one currently absent from
the public ecosystem (confirmed via direct GitHub code-search +
ecosystem scan during Phase B research streams):

1. **First public OSCAL serialization of OpenSSF OSPS Baseline**
   (no prior published; Apache-2.0 = zero redistribution friction)
2. **First public `OSPS-CONFORMANCE.md` self-attestation** (zero
   results on `gh api search/code "OSPS-CONFORMANCE.md"`)
3. **First Apache-2.0 machine-readable OSS DORA Art. 17 incident
   emitter scaffolding** (preview shape; full library lands v0.11)
4. **First Tier-A OSPS-Baseline bundled control catalog set** in
   any GRC tool (commercial GRC vendors have not surfaced OSPS
   coverage)

Together these widen Evidentia's "OSS + OSCAL + air-gap + eval-
rigor" uncontested quadrant rather than narrow it (per Phase B
audit Theme T1 + competitive landscape DELTA in Stream E4).

## 3. Phases

### Phase 1 — OSPS Baseline bundled catalog (Tier-A, 3 maturity files)

- **Source**: `github.com/ossf/security-baseline/baseline/`
  (Apache-2.0; 8 family YAMLs + 41 controls + 3 maturity levels)
- **Catalog files** (in `packages/evidentia-core/src/evidentia_core/catalogs/data/international/`):
  - `osps-baseline-m1.yaml` (21 controls — any project)
  - `osps-baseline-m2.yaml` (38 controls — ≥2 maintainers + small
    consistent user base; with solo-dev honest-gap footnote)
  - `osps-baseline-m3.yaml` (58 controls — large consistent user
    base; partial coverage; honest-gap declarations for OSPS-AC-04,
    OSPS-GV-04, OSPS-QA-07 structurally-unreachable-solo controls)
- **Pattern precedent**: matches existing `fedramp-rev5-{low,
  moderate,high}.yaml` + `cmmc-2-{l1,l2,l3}.yaml` 3-catalog
  flatten pattern; zero schema change.
- **Crosswalk slot**: each catalog file documents that `guidelines[]`
  in upstream OSPS YAMLs map to 10 external frameworks (NIST SSDF,
  NIST CSF, EU CRA, PCI DSS, NIST 800-161, etc.) — formal crosswalks
  ship in v0.10.6 Phase 1.
- **Effort**: ~3-4 hours for the 3 YAML files; ~700 lines total.
- **License posture**: Tier A (Apache-2.0 verbatim bundling).

### Phase 2 — OSCAL conversion of OSPS Baseline (first-mover)

- **Target file**: `packages/evidentia-core/src/evidentia_core/catalogs/data/international/osps-baseline.oscal.json`
- **Schema**: NIST OSCAL Catalog v1.2.1 (Evidentia's current).
- **Process**: clean-room transform from Phase 1 YAMLs to OSCAL
  Catalog shape; preserve every OSPS control + assessment-requirement
  + maturity-level applicability per the OSCAL `props` extension
  pattern.
- **Effort**: ~3-4 hours.
- **Upstream PR (Phase 2.A)**: submit conversion to
  `oscal-club/awesome-oscal` as the **first OSCAL serialization of
  an OpenSSF flagship deliverable** (zero prior); separately offer
  to `ossf/security-baseline` as a contribution. **Both PRs are
  separate publishing actions requiring per-action approval per
  `~/.claude/CLAUDE.md` publishing-authority protocol.**

### Phase 3 — OSPS-CONFORMANCE.md + osps-conformance.yaml + CI gate

- **Repo-root file**: `OSPS-CONFORMANCE.md` (~300 lines).
- **Claim shape**: "As of <date>, Polycentric-Labs/evidentia
  complies with OSPS Baseline v2026.02.19 Level 2 (with partial
  Level 3 coverage). Self-assessed. No third-party audit."
- **Per-control evidence table**: 21 Level 1 PASS + 17 Level 2
  PASS + partial Level 3 with cited evidence (commit SHA / file
  path / `gh api` query / CI workflow URL) per control.
- **Honest-gap section**: 3 structurally-unreachable-solo controls
  (OSPS-AC-04.01/02 explicit CI/CD permissions audit, OSPS-GV-04
  formal permission-grant review, OSPS-QA-07 two-person review)
  declared as honest-gaps tied to the commercial-tier hiring
  milestone per `evidentia_commercial_strategy.md`.
- **Machine-readable companion**: `.local/pre-release-review/osps-conformance.yaml` (skill
  v5.1 first-run-bootstrap pattern; per-control PASS|FAIL|HONEST_GAP
  + evidence pointers).
- **CI gate**: new workflow `verify-osps-conformance.yml` that
  re-validates each claimed-PASS control's evidence link via
  `gh api` calls; fails if any cited evidence 404s.
- **First-mover claim**: `gh api search/code "OSPS-CONFORMANCE.md"`
  returns 0 (verified Phase B Stream E4 2026-05-24). Evidentia
  would be the first public open-source project to ship this

- **Effort**: ~1-2 days for doc + CI workflow.

### Phase 9 — `evidentia-eval` workspace package extraction (Kimi audit close-out)

- Extract DFAH determinism + faithfulness harness from
  `evidentia-ai/eval/` to a new standalone workspace package
  `packages/evidentia-eval/`. Same symbols, same signatures
  (`DFAHarness`, `EvalResult`, `EvalSample`, `DeterminismResult`,
  `ReplayResult`, `FaithfulnessResult`,
  `PromptFaithfulnessResult`, `faithfulness_score`,
  `faithfulness_score_semantic`, `extract_claims`,
  `normalize_for_determinism`, `hash_output`, `sign_eval_result`,
  `verify_eval_result`) — only the import path changes.
- **Rationale**: air-gap installs of the production risk-statement
  runtime (a key Evidentia positioning differentiator) should not
  transitively pull the dev-time eval stack
  (sentence-transformers + numpy + claim-extraction LLM call
  surface). The Kimi engineering audit flagged that
  `import evidentia_ai` *might* top-level-load torch /
  transformers — verified in this phase to be **refuted** (already
  lazy); the extraction makes the install-size separation a
  hard contract instead of a soft convention.
- **Backward-compat shim**: `evidentia_ai.eval.*` remains as a
  deprecation shim through v0.11.x. Importing fires a
  `DeprecationWarning` and re-exports from `evidentia_eval.*`.
  Removal scheduled for v0.12.0 (2-minor-version migration
  window per `docs/api-stability.md` §1).
- **Dep reshuffling**: `evidentia-ai[eval-faithfulness]` now
  proxies to `evidentia-eval[faithfulness-semantic]`. Same
  underlying `sentence-transformers >= 3.0` + `numpy >= 1.26`
  stack — the heavy deps just live on the eval package now.
- **New test**: `tests/unit/test_ai/test_lazy_imports.py` pins
  the lazy-import posture in CI via subprocess assertions —
  guards against a future refactor accidentally regressing the
  air-gap-clean install path.
- **bump_version.py**: no script edit needed. The v0.10.1
  F-V100-M1 fix reads `[tool.uv.sources]` dynamically as the
  workspace allowlist; adding
  `evidentia-eval = { workspace = true }` to the root pyproject
  automatically includes evidentia-eval in version-bump pin
  substitution (verified via dry-run: 8 workspace pkgs detected).
- **Effort**: ~3-4 hours (file moves + import rewrites + shim
  authoring + dep reshuffling + lazy-import test + docs).
- **Validation gates passed**: 3396 tests / 0 failures / 17
  skipped (was 3370; +26 from new lazy-import tests +
  refactored mock paths); mypy strict 0/0 across 276 source
  files (was 268); ruff clean; `uv sync --all-packages` clean.
